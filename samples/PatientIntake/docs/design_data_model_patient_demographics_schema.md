# Patient Demographics Table Schema

## Architecture Overview for Patient Intake System

1. System Context
 The system is a SaaS‑style patient intake platform deployed on‑prem via Docker Compose but exposed as a multi‑tenant web service. It ingests PHI through a WCAG‑compliant web form, stores encrypted records in PostgreSQL, and provides PDF export for authorized roles.

2. Component Diagram
 - Presentation Layer: React SPA served by Nginx (static). Communicates over HTTPS with API Gateway.
 - API Gateway: FastAPI (Python) handling `/api/v1/*` routes, JWT RS256 authentication, request validation via Pydantic.
 - Service Layer: PatientService, AuditService, PdfService implemented as separate FastAPI routers.
 - Data Layer: PostgreSQL 13 with pgcrypto for AES‑256‑GCM field‑level encryption and Row‑Level Security policies.
 - Security Services: Vault for key storage, Docker secrets for DB credentials.
 - PDF Generation: WeasyPrint invoked by PdfService to produce watermarked PDFs.

3. Interaction Flow
 1. Browser loads SPA over TLS.
 2. User submits form → POST `/api/v1/patients` with encrypted payload.
 3. API validates JWT, stores encrypted fields, creates audit entry.
 4. Authorized clinician requests PDF → GET `/api/v1/patients/{id}/export`.
 5. PdfService fetches decrypted data, renders HTML template, calls WeasyPrint, adds watermark with user ID and timestamp, returns PDF stream.

4. API Endpoints

| `/api/v1/patients` | POST | Create patient record | `{ "first_name": "string", "last_name": "string", "dob": "date", "ssN_encrypted": "string", "insurance": { "provider": "string", "policy_number_encrypted": "string" } }` | `{ "patient_id": "uuid", "status": "created" }` | Bearer JWT |

8. Rate Limiting & Performance Constraints
- Global API rate limit: 100 requests per minute per authenticated user (KPI‑043). Exceeding returns ERR-007 with `Retry-After` header.
- Form submission latency target: ≤200 ms p95 (KPI‑02).
- PDF generation latency target: ≤200 ms p95 (KPI‑050).
- Audit log write latency target: <100 ms (KPI‑042).

9. Audit Logging (Patient & Medical History)
Every successful POST, GET, DELETE, and PDF export triggers an INSERT into `audit_log` with fields: `log_id` (UUID), `user_id` (from JWT), `action` (e.g., `CREATE_PATIENT`, `GET_MEDICAL_HISTORY`), `resource` (endpoint path), `resource_id`, `timestamp`, `outcome` (`success`/`failure`). Failures to write the audit log cause the transaction to roll back and return ERR-006.

10. Security Details
- JWT must be signed with RS256; public keys are fetched from Vault at startup. Missing or malformed keys result in ERR-002.
- All secrets (DB credentials, encryption keys) are provided via Docker secrets; absence triggers ERR-002.
- Row‑Level Security policies enforce FR-002 (role‑based access) and FR-003 (audit logging).

11. Traceability
- FR-001 – Form submission latency ≤200 ms (referenced in section 8).
- FR-002 – Role‑based access control enforced via RLS (section 10).
- FR-003 – All actions logged as per section 9.
- REQ-001 – WCAG 2.1 AA compliance for form fields (section 1).
- REQ-002 – Keyboard navigation support (section 1).
- KPI‑042 – Audit log write latency target (section 8).
- KPI‑043 – Rate limit compliance (section 8).

## Medical History Service Design

### 1. Overview
The Medical History service stores patient‑provided clinical information required for care coordination while satisfying HIPAA technical safeguard requirements. All PHI fields are encrypted at rest using pgcrypto AES‑256‑GCM and transmitted over TLS\u00a01.3. The design includes row‑level security (RLS), audit logging, rate limiting, pagination, and comprehensive error handling.

### 2. Audit Logging Integration
A trigger `audit_medical_history` writes an entry to `audit_log` on INSERT, UPDATE, DELETE. The log captures `user_id`, `action`, `table_name`, `row_id`, `timestamp`, and a SHA‑256 hash of the changed data for tamper evidence. API operations also emit audit events via the `AuditService` using PostgreSQL LISTEN/NOTIFY.

### 3. Encryption Details
* Encryption key stored in Docker secret `pgcrypto_key`.
* Field‑level encryption performed with `pgp_sym_encrypt(data, current_setting('app.pgcrypto_key'))`.
* Decryption with `pgp_sym_decrypt`.
* All encrypted columns are defined NOT NULL after encryption to avoid plaintext leakage.
* Meets REQ‑006 (AES‑256‑GCM) and REQ‑007 (key rotation process).

### 4. Indexes for Performance
* Primary key index on `id`.
* B‑tree index on `patient_id` for fast joins (supports latency requirement in FR‑001).
* GIN index on `chronic_conditions` array for condition‑based queries.

## Technical Design Specification

### 10. Service Descriptions
| Service | Responsibilities | Dependencies | Produced Events | Consumed Events |
|---|---|---|---|---|
| History API Service (SVC‑001) | CRUD for `medical_history`; enforce RLS; encrypt PHI fields | PostgreSQL, Auth Service (JWT verification) | history.created, history.updated, history.deleted | auth.success, audit.log_written |
| PDF Generator Service (SVC‑002) | Assemble patient PDF, apply watermark/timestamp | History API Service (internal), WeasyPrint, Docker secret for signing key | pdf.generated | history.requested |
| Audit Service (SVC‑003) | Capture all data‑access events; write immutable logs | PostgreSQL, RabbitMQ | audit.log_written | db.operation, history.created, pdf.generated |

#### 10.1 `medical_history` Table
sql
CREATE TABLE medical_history (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    patient_id UUID NOT NULL REFERENCES patients(id),
    entry_date TIMESTAMPTZ NOT NULL DEFAULT now(),
    diagnosis TEXT NOT NULL ENCRYPTED USING pgp_sym_encrypt(diagnosis, current_setting('app.crypto_key')),
    treatment TEXT ENCRYPTED USING pgp_sym_encrypt(treatment, current_setting('app.crypto_key')),
    created_by UUID NOT NULL,
    updated_by UUID,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ
);

RLS policy ensures clinicians can only access rows where `patient_id` belongs to their assigned cohort.

#### 10.2 `audit_log` Table
sql
CREATE TABLE audit_log (
    id BIGSERIAL PRIMARY KEY,
    operation TEXT NOT NULL,
    table_name TEXT NOT NULL,
    record_id UUID NOT NULL,
    user_id UUID NOT NULL,
    timestamp TIMESTAMPTZ NOT NULL DEFAULT now(),
    details JSONB NOT NULL
);

All INSERT/UPDATE/DELETE on `medical_history` trigger `pg_notify('audit', json_build_object(...))` to guarantee immutability.

#### 11.2 PDF Generator Endpoint
**GET** `/api/v1/patients/{pid}/export` – Returns PDF stream.
Request headers: `Accept: application/pdf`, optional `Range` for partial download.
Success: `200 OK` with `Content-Type: application/pdf`. Errors follow the generic error schema; additionally `503 Service Unavailable` when generator is down (with `Retry-After`). Rate limit: 30 exports/min per user (KPI‑042).

### 13. Integration Flow (Event‑Driven)
1. Front‑end posts JSON to History API (`POST /api/v1/patients/{pid}/history`).
2. Middleware validates JWT, sets RLS context, encrypts PHI fields and inserts row.
3. On commit, a `history.created` event is published to RabbitMQ.
4. Audit Service consumes the event, writes an immutable entry to `audit_log`.
5. Clinician requests PDF (`GET /api/v1/patients/{pid}/export`).
6. PDF Generator calls History API (`GET /api/v1/patients/{pid}/history`) internally, decrypts fields using secret key, renders HTML and generates PDF via WeasyPrint. Watermark includes clinician user‑ID and ISO‑8601 timestamp.
7. Generator publishes `pdf.generated`; Audit Service logs the export action.
8. All services emit health metrics to Prometheus for monitoring latency KPIs (KPI‑001).

### 14. Performance & Rate‑Limiting Constraints
- **Form Submission Latency** – Target ≤200 ms for History API write (KPI‑001). Measured at the Nginx edge.
- **PDF Generation Latency** – ≤200 ms for PDF creation after data retrieval (KPI‑050).
- **Rate Limits** – 100 requests/min/user for History API; 30 exports/min/user for PDF Generator. Exceeding returns `429 Too Many Requests` with `Retry-After`.
- **Pagination** – Default page size 20; max page size 50 to bound response payloads.

### 15. Monitoring & Observability
- **Metrics** – Prometheus counters for request rates, error rates, latency histograms per endpoint. Alerts fire on >95th percentile latency breach or error rate >1 %.
- **Logs** – Structured JSON logs shipped to ELK stack; audit logs indexed separately for compliance queries.
- **Health Checks** – `/healthz` endpoint on each service returns readiness/liveness status for Kubernetes or Docker Compose orchestrator.
