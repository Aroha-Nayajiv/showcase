# Patient Data Access API Contract

## 1. System Overview
The architecture supports a HIPAA‑compliant patient intake system deployed on‑prem via Docker Compose. It consists of four logical layers: Front‑end Web UI, API Gateway, Service Layer (Auth Service, Patient Service, Audit Service), and PostgreSQL database with pgcrypto encryption. All inter‑service communication uses HTTPS with TLS 1.3 and JWT RS256 tokens issued by the Auth Service.

## 2. Component Diagram
Web UI (React, served by Nginx) → calls API Gateway.
API Gateway (Traefik) → routes to micro‑services, enforces rate limiting and TLS termination.
Auth Service → validates credentials, issues JWT containing role claim (admin, clinician, front‑desk).
Patient Service → implements CRUD endpoints defined in this contract, accesses encrypted columns via pgcrypto.
Audit Service → writes immutable audit records to PostgreSQL table audit_log using append‑only strategy; logs include user_id, timestamp, action, patient_id.
PostgreSQL (12+, pgcrypto enabled) → stores patients table with column‑level encryption for PHI fields, users table for role mapping, and audit_log table.

## 3. Deployment Topology
Docker Compose defines services: frontend, gateway, auth, patient, audit, db. Each service runs in its own container with limited capabilities; the database container mounts a volume for data persistence and uses Docker secret POSTGRES_PASSWORD_FILE. No external cloud services are referenced.

## 4. Scalability and High Availability
Stateless services allow horizontal scaling behind the gateway.
PostgreSQL can be replicated using streaming replication; read replicas serve audit queries.
Health checks and automatic restart policies defined in Docker Compose.

## 5. Failure Handling
If Auth Service is unavailable, gateway returns 503 Service Unavailable; clients must retry with exponential backoff.
If database connection fails, services fallback to cached read‑only mode for non‑critical metadata and log the incident.
Audit Service failures trigger a retry queue; after three attempts the request is persisted to a local file for later bulk upload.

## 6. Monitoring and Alerts
Prometheus scrapes metrics from each container; alerts fire on:
- JWT validation errors > 5/min (possible attack).
- Audit write latency > 100 ms (KPI‑042).
- Database CPU > 80% sustained (scale trigger).

All error responses follow the schema:

Specific error codes are defined in Section 7.4.

#### 7.2.2 Acceptance Criteria
- Successful POST /api/v1/patients must respond within 200 ms (KPI‑02).
- GET /api/v1/patients/{id} must return 404 if record not found.
- PDF export must embed watermark containing user ID and ISO‑8601 timestamp (FR‑008) and complete within 500 ms.
- All endpoints must log request and response metadata to Audit Service (FR‑003).

### 7.6 Security Considerations
- **Authentication**: RS256 JWT validated at API gateway; token includes `sub`, `role`, `exp` claims.
- **Authorization**: Role‑based access control enforced per endpoint; clinicians may only read patients assigned to them via PostgreSQL row‑level security policies.
- **Transport Encryption**: All traffic terminated at Nginx with TLS 1.3 using certificates mounted as Docker secrets.
- **At‑Rest Encryption**: Sensitive fields (`medical_history_encrypted`) stored using pgcrypto AES‑256‑GCM; encryption keys loaded from Docker secret `pgcrypto_key` and rotated quarterly per REQ‑007.
- **Input Validation**: FastAPI JSON schema validation; regex constraints for identifiers; date fields validated against ISO‑8601.
- **Audit Logging**: Every successful and failed request writes an immutable log entry containing `user_id`, `action`, `resource_id`, `timestamp`, and `outcome` to `audit_log` table.
- **Failure Handling**: Encryption Service unavailability returns ERR‑005 with `retryable:true`; client should implement exponential back‑off.

## Functional Requirements
- FR-001: Clinicians must be able to view patient intake records within 2 seconds of submission (p95 response time).
- FR-002: Access to patient history must be restricted to records of patients assigned to the clinician (RBAC via JWT claims and PostgreSQL RLS).
- FR-003: All view actions must be logged with user ID, timestamp, and record ID; logs retained for 7 years on immutable storage (KPI-003).
- FR-008: System must support export of patient intake summaries as PDF with watermark containing export timestamp and exporting user ID (KPI-030).

### Tables
1. **patient**
   - id UUID PRIMARY KEY (gen_random_uuid())
   - first_name TEXT ENCRYPTED (pgp_sym_encrypt)
   - last_name TEXT ENCRYPTED
   - date_of_birth DATE ENCRYPTED
   - gender TEXT ENCRYPTED
   - phone_number TEXT ENCRYPTED
   - email TEXT ENCRYPTED UNIQUE INDEX
2. **user_account**
   - id UUID PRIMARY KEY
   - username TEXT UNIQUE
   - password_hash TEXT
   - role_id UUID REFERENCES role(id)
3. **role**
   - id UUID PRIMARY KEY
   - name TEXT UNIQUE (e.g., clinician, admin)
4. **audit_log**
   - id BIGSERIAL PRIMARY KEY
   - user_id UUID REFERENCES user_account(id)
   - patient_id UUID REFERENCES patient(id)
   - action TEXT
   - timestamp TIMESTAMPTZ DEFAULT now()
   - details JSONB
   - INDEX idx_audit_user (user_id)
   - INDEX idx_audit_patient (patient_id)

## API Contracts

### Authentication Service (Auth)
- **POST /api/v1/auth/login**
  Request: `{"username":"string","password":"string"}`
  Response (200): `{"access_token":"<JWT>","token_type":"Bearer"}`
  Errors: 401 ERR-001 (Invalid credentials).
  JWT Claims: `sub` (user_id), `role` (role name), `exp`, `iat`. Signed with RS256 using key from Docker secret `/run/secrets/jwt_private_key`.

### Patient Data Access API
- **GET /api/patients/{id}**
  Auth: Bearer JWT required.
  Response (200): `{"id":"uuid","first_name":"string","last_name":"string","date_of_birth":"YYYY-MM-DD","gender":"M|F|Other","phone_number":"string","email":"string"}` (fields decrypted in service layer).
  Errors: 401 ERR-001 (Unauthenticated), 403 ERR-002 (Access denied by RLS), 404 ERR-003 (Not found), 500 ERR-005 (Internal server error).

- **POST /api/patients**
  Request body: same fields as GET (except id).
  Response (201): `{"id":"uuid"}`.
  Errors: 400 ERR-004 (Validation error), 401 ERR-001, 403 ERR-002, 500 ERR-005.

## Authentication & Authorization Details
- JWT signed with RS256 using private key stored as Docker secret `/run/secrets/jwt_private_key`. Public key exposed via `/jwks.json`.
- PostgreSQL Row‑Level Security policies enforce that clinicians can only access rows where `current_setting('app.current_user_id')` matches `patient.owner_id` and role is `clinician`. Admins have unrestricted access. Policies are defined in `patient_rls.sql`.

## Encryption & Key Management
- All PHI columns are encrypted at rest using `pgp_sym_encrypt(column, dearmor('master_key'))` where `master_key` is loaded from Docker secret `/run/secrets/pg_master_key`.
- Key rotation procedure: generate new secret, update Docker secret, run migration script to re‑encrypt existing data using `pgp_sym_encrypt` with new key while preserving old key for decryption during transition. Documentation added in `key_rotation.md`.

## Audit Logging Specification
- Service `AuditService` writes an entry for every API request that modifies or reads PHI. Entry schema matches `audit_log` table described above.
- Log retention: immutable storage for 7 years; archival to cold storage after 2 years.
- KPI‑003 measured via monitoring query counting rows inserted per minute; alert threshold >95th percentile latency >100 ms.

## PDF Export Service Details
- PDF generated using wkhtmltopdf container. Watermark text composed as `"Exported by {user_id} at {timestamp}"` where `timestamp` is ISO‑8601 UTC. Watermark placed in footer on each page.
- PDF generation latency target <200 ms (KPI‑046). Errors return JSON `{ "error":"PDF generation failed","code":"ERR-005"}` with HTTP 500.
