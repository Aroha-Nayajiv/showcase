# Insurance Data Entry Journey Flow

### Persona 1: Front‑Desk Staff (Intake Clerk)
Primary Goal: Capture complete patient demographics, insurance information, and medical history quickly and accurately.
Key Tasks: Open web form, enter data, validate fields, submit, receive confirmation receipt.
Compliance Requirements: Must not view PHI beyond entry scope; field‑level encryption active; trigger audit log entry (FR‑003).
Success Metrics: Form submission latency <200 ms (KPI‑002), error‑rate <1 % (FR‑005), receipt displayed within 1 s (FR‑006).

### Persona 3: Administrator (System Admin / Compliance Officer)
Primary Goal: Configure RBAC, monitor audit logs, ensure HIPAA compliance.
Key Tasks: Define PostgreSQL roles, set row‑level security policies, review audit logs, generate compliance reports, manage Docker Compose deployment.
Compliance Requirements: Enforce RBAC per FR‑002; maintain immutable audit logs for 7 years (FR‑003, KPI‑017); encryption key rotation per NFR‑001 and HIPAA §164.312(a)(2)(iv).
Success Metrics: No unauthorized access incidents; audit log completeness 100 % (KPI‑003); deployment reproducible without external cloud services (FR‑009).

## Edge Cases Summary
- Missing encryption key – system fails safe and logs error.
- Unauthorized PDF access – access denied and logged.
- Resource exhaustion during export – graceful degradation with logged failure.
- Concurrent edits – last‑write‑wins with version check.
- Network interruption during form submit – client retries up to three times.

## Design Needs Derived from Personas
- **Encryption Specification**: Field‑level encryption using pgcrypto; TLS 1.3 for in‑transit protection; PDFs encrypted at rest AES‑256.
- **RBAC Model**: PostgreSQL roles `front_desk`, `clinician`, `admin` with row‑level security policies referencing `current_user`.
- **Audit Log Format**: Immutable JSON entries containing `user_id`, `action`, `timestamp`, `record_id`, `outcome`.
- **Performance Targets**: Form response <200 ms; PDF generation ≤2 s; receipt download ≤1 s.
- **Compliance Checks**: Field‑level encryption validation (FR‑001); RBAC enforcement (FR‑002); immutable audit logs retention 7 years (FR‑003).

## PDF Summary Feature Specification - MVP

### 1. Prioritized User Stories

- **US-001** *Clinician*: Generate a PDF summary of a patient's intake data. *Goal*: Review information offline while maintaining auditability.
  - **Acceptance Criteria**:
    - **Given** a clinician is authenticated and has access to patient Y’s intake record,
    - **When** the clinician selects “Export PDF”,
    - **Then** the system generates a PDF within 2 seconds, includes a watermark "Exported by {UserID} on {ISO‑8601 timestamp}" and stores the file encrypted at rest (AES‑256), and logs the export event (FR‑003, KPI‑030).
- **US-002** *Front‑desk staff*: Export a PDF receipt after completing a patient’s intake form.
  - **Acceptance Criteria**:
    - **Given** the intake form has passed validation,
    - **When** the staff clicks “Generate Receipt PDF”,
    - **Then** a PDF receipt containing patient name, submission timestamp, unique receipt ID and the watermark "Generated by Front Desk" is created within 2 seconds, encrypted at rest, and a download completes within 1 second. If validation failed, the system shows error "Cannot generate receipt for invalid submission".
- **US-003** *Administrator*: Configure role‑based permissions for PDF export.
  - **Acceptance Criteria**:
    - **Given** an admin is authenticated,
    - **When** the admin updates the RBAC policy for PDF export,
    - **Then** only users with the "clinician" or "front_desk" role can invoke the PDF export endpoint, and attempts by unauthorized roles are denied and logged.

### 2. Success Metrics

- PDF generation latency ≤ 2 seconds for 95 % of requests (KPI‑030).
- Watermark accuracy 100 % (user ID and ISO‑8601 timestamp present) (KPI‑030).
- Audit log completeness 100 % of export actions recorded (FR‑003).

### 3. Design Needs for Designers

- **Watermark Engine**: Use an open‑source library (e.g., pdf‑lib for Node.js or ReportLab for Python) to embed dynamic text footer "Exported by {UserID} on {YYYY‑MM‑DD HH:MM:SS UTC}" with 30 % opacity and also store the same string in PDF metadata (invisible).
- **Encryption**: PDFs must be encrypted at rest with AES‑256 (pdf‑lib custom wrapper) and stored in a directory with permissions `chmod 600` owned by the service account.
- **Transport Security**: All PDF generation and download traffic must use TLS 1.3.
- **Audit Logging**: Export events must emit a log entry conforming to NIST SP 800‑53 AU‑12 format, containing operation, actor, patient ID, timestamp, outcome, and file hash. Logs are immutable for 7 years (KPI‑003).
- **Performance Monitoring**: Expose Prometheus metric `pdf_generation_seconds` with alert if >2 s for >5 % of requests.

### 4. API Specification (Design Reference Only)

- `POST /api/v1/patients/{patient_id}/pdf/export`
  - **Headers**: `Authorization: Bearer <token>`
  - **Responses**:
    - `200 OK` – body: binary PDF, `Content-Type: application/pdf`
    - `400 Bad Request` – error "Cannot generate receipt for invalid submission"
    - `403 Forbidden` – error "User not authorized to export PDF"
    - `500 Internal Server Error` – generic error message.

### 5. Dependencies & Assumptions

- PostgreSQL row‑level security is configured for `clinician` and `front_desk` roles (FR‑002).
- Field‑level encryption at rest is already in place (FR‑001).
- Docker Compose provides a tmpfs volume for temporary PDF files to avoid plaintext persistence.
- HashiCorp Vault manages encryption keys; keys rotated every 90 days (mitigates RISK‑006).

### 6. MVP Scope Summary

The MVP delivers US‑001, US‑002, and US‑003 with full acceptance criteria, watermarking, AES‑256 at‑rest encryption, TLS 1.3 transport, audit logging, and performance monitoring. Out‑of‑scope items: batch export, custom branding, digital signatures, and multi‑language PDFs.
