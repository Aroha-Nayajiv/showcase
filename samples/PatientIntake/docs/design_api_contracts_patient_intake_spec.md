# Patient Intake API Contract

## System Architecture Overview
1. Presentation Layer
The web client is a React SPA served over HTTPS (TLS 1.3). It renders the patient intake form, performs client‑side validation, and encrypts PHI fields using the open‑source AES‑256‑GCM envelope encryption library before transmission. The UI complies with WCAG 2.1 AA (REQ‑001, REQ‑002) and provides real‑time error feedback to keep the validation error rate below 1 % (KPI‑005).
2. API Gateway
All external calls pass through an Nginx reverse‑proxy configured for TLS termination and rate limiting (max 200 req/s per IP). The gateway forwards requests to the Patient Intake Service using HTTP 2.0 and validates JWT RS256 tokens (REQ‑007). Unauthorized requests receive ERR‑401 (Unauthorized) with a non‑retryable flag.
3. Service Layer
The Patient Intake Service implements business rules, orchestrates PDF generation, and emits domain events. It exposes three REST endpoints under the /api/v1/ prefix:
   • POST /api/v1/patients – creates a new intake record.
   • GET /api/v1/patients/{id} – retrieves a patient record for authorized roles.
   • GET /api/v1/patients/{id}/summary – returns a PDF generated by WeasyPrint with a timestamped watermark (FR‑008).
All endpoints require a Bearer token and enforce role‑based access control (admin, clinician, front‑desk) via PostgreSQL row‑level security policies defined in the data model.
4. Data Persistence
PostgreSQL 15 runs in a Docker container with pgcrypto enabled for field‑level encryption at rest (AES‑256‑GCM). Tables: patients, demographics, insurance, medical_history, audit_log, rls_roles. Each PHI column is stored encrypted using pgp_sym_encrypt with a per‑field data‑encryption key wrapped by a master key stored in a Docker secret (POSTGRES_PASSWORD_FILE). Audit logs capture every INSERT, UPDATE, DELETE, and SELECT with user_id, timestamp, and operation type (FR‑003). Indexes on patient_id and created_at ensure query latency below 200 ms (KPI‑02).
5. Security Controls
   • Transport security: TLS 1.3 with strong cipher suites.
   • Authentication: RSA‑based JWT RS256 signed by an internal key pair; public key distributed via JWKS endpoint.
   • Authorization: PostgreSQL RLS policies map JWT role claims to row visibility.
   • Auditing: log_statement = 'all' and pg_audit extension generate immutable audit records retained for 7 years (KPI‑003).
   • Key Management: Master key rotated monthly via Docker secret update; rotation logged as audit event.
6. Deployment & Operations
Docker Compose defines three services: web (React), api (Python FastAPI), db (PostgreSQL). All containers run with the --read‑only flag (FR‑017) and use Docker secrets for credentials. Healthchecks restart the API if the DB becomes unavailable; the API returns ERR‑503 (Service Unavailable) with retryable true.
7. Compliance Traceability
Each architectural decision references a requirement ID: FR‑001, FR‑002, REQ‑001, REQ‑006, KPI‑001, KPI‑003, etc., enabling auditors to verify HIPAA technical safeguard compliance.

## Security Controls (expanded)
- **Transport Layer**: Enforce TLS 1.3 with cipher suite `TLS_AES_256_GCM_SHA384`.
- **Authentication**: JWT signed with RSA‑256; keys rotated quarterly; JWKS endpoint `/jwks.json`.
- **Authorization**: PostgreSQL roles `admin_role`, `clinician_role`, `frontdesk_role` mapped via RLS policies (`CREATE POLICY patient_view ON patients USING (jwt_claims->>'role' = 'clinician' AND patient_id = ANY (SELECT patient_id FROM assignments WHERE user_id = current_user_id()))`).
- **Encryption at Rest**: pgcrypto `pgp_sym_encrypt` using per‑field DEK wrapped by master key stored in Docker secret `master_key`.
- **Key Management**: Master key rotated monthly via Docker secret update; rotation logged as audit event `key_rotation`.
- **Audit Logging**: `pg_audit` captures all DML; logs stored in immutable S3 bucket (retention 7 years).
- **Rate Limiting**: Nginx `limit_req zone=api burst=20 nodelay`.
- **Container Hardening**: All containers run with `--read-only`; only `/tmp` mounted as tmpfs; capabilities dropped except `CAP_NET_BIND_SERVICE`.

## Technical Design – Patient Intake System

### 1. Overview
The Patient Intake system enables healthcare staff to capture, store, and retrieve patient demographic, insurance, and medical history data while meeting HIPAA, performance, and auditability requirements (FR-001‑FR-010, REQ-001, REQ-002, REQ-007). All services are containerized and orchestrated with Docker Compose (FR-009).

#### Relationships
- patients 1‑1 patient_demographics (patient_id)
- patients 1‑many patient_insurance (patient_id)
- patients 1‑many patient_medical_history (patient_id)
- app_users many‑to‑1 app_roles (role_name)
All foreign keys use `ON DELETE RESTRICT` to preserve auditability.

### 3. Service Layer Interfaces

### Audit Logging & Monitoring
- Every SELECT/INSERT/UPDATE/DELETE on PHI triggers an entry in `audit_log`.
- Logical replication streams logs to Elasticsearch; retention 7 years satisfies KPI‑003.
- Alerts for TLS handshake failures, repeated auth errors, and anomalous audit patterns are sent to Prometheus Alertmanager and PagerDuty.
