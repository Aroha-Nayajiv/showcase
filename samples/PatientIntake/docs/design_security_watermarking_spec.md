# PDF Watermarking Specification

## High-Level System Architecture Diagram
1. Presentation Layer
The presentation layer is implemented with React and TypeScript compiled to static assets served by an Nginx container. Nginx terminates TLS using certificates generated by the internal PKI for the air‑gap environment. All client‑to‑server traffic is encrypted to satisfy 45 CFR 164.312(a)(2)(iv) encryption at transit.
2. API Layer
The API layer consists of a FastAPI application written in Python 3.11. All endpoints are versioned under the /api/v1/ prefix and enforce Bearer token authentication using RS256 signed JWTs as defined in the project design. The API gateway runs in its own Docker service and validates tokens against the public key stored in a Docker secret.
3. Service Layer
a. PDF Generation Service – implemented with WeasyPrint to convert HTML templates into PDF files. The service adds a watermark containing the export timestamp and the authenticated user identifier. Performance target is average generation time below 200 ms per document.
b. Audit Logging Service – writes immutable audit records to PostgreSQL using INSERT statements wrapped in a transaction. Log entries include user ID, action type, timestamp, and source IP. Write latency must remain under 100 ms to meet KPI‑042.
c. Validation Service – performs server‑side form validation and field‑level encryption before persisting data.
4. Data Layer
PostgreSQL 15 runs in a dedicated container with pgcrypto extension enabled. Sensitive fields such as social security number and medical history are encrypted at rest using AES‑256‑GCM via pgcrypto's encrypt function. Row‑level security policies restrict row visibility based on the role attribute (admin, clinician, front_desk). Indexes on patient_id and created_at support fast lookup for audit queries.
5. Integration Components
Docker Compose orchestrates all containers, defines a private overlay network, and sets each container file system to read‑only where possible (Docker flag --read-only). Secrets such as POSTGRES_PASSWORD are injected via Docker secrets mounted at /run/secrets/. Healthcheck probes monitor container liveness every 30 seconds and trigger automatic restart on failure.
6. Security Controls
In addition to TLS and field‑level encryption, the system enforces principle of least privilege by granting each service only the database role required for its function. Audit logs are retained for seven years in immutable storage as required by FR‑003 and KPI‑003.
7. Compliance Mapping
All design decisions map to HIPAA technical safeguards: encryption at rest and in transit (45 CFR 164.312(a)(2)(iv)), audit controls (45 CFR 164.312(b)), and access control (45 CFR 164.312(a)(1)). The architecture also satisfies SOC 2 criteria for confidentiality and integrity.
8. Scalability and Availability
The Nginx front end can be scaled horizontally by adding replicas behind a Docker swarm load balancer. PostgreSQL uses streaming replication for high availability, and failover is orchestrated by Patroni within the same compose file.
9. Monitoring
Prometheus scrapes metrics from FastAPI, WeasyPrint, and PostgreSQL exporters. Alerts fire on response time exceeding 250 ms or audit log write latency exceeding 150 ms.
10. Documentation
All contracts, schemas, and error codes are versioned in the repository under /docs/contracts/. This diagram serves as the single source of truth for downstream development and testing phases.

### 1. Purpose
This document defines the concrete HTTP contracts that the PatientIntake system will expose for collecting patient demographics, insurance information, and medical history. All endpoints are versioned under the `/api/v1/` prefix, use JSON payloads, and enforce HIPAA‑required encryption in transit via TLS 1.3. The contracts are ready for implementation by the Development team.

### 2. Authentication & Authorization
All requests must include a Bearer token issued by the `/api/v1/auth/login` endpoint. Tokens are JWTs signed with RS256 as mandated by the design artifacts. Role‑based access control (RBAC) is enforced per endpoint: admin and clinician may POST new intake records; front_desk may only POST basic demographics and insurance but not medical history. The token payload contains a role claim that the service validates against the required role matrix.

## Design PostgreSQL Data Model (SCH-002)

### Overview
The PatientIntake system stores protected health information (PHI) in a PostgreSQL database hardened for HIPAA compliance. All tables are created with column‑level encryption using pgcrypto's pgp_sym_encrypt wrapper (AES‑256‑GCM) and row‑level security (RLS) policies that enforce role‑based access control (admin, clinician, front_desk). The schema is versioned via a schema_migrations table to support incremental upgrades.

### Entity Definitions
Core entities: patient, insurance, medical_history, user_role, audit_log, and supporting lookup tables. Each entity includes a UUID primary key, audit columns (created_at, updated_at), and encrypted PHI fields where required.

### Table Specifications
Patient table (patients) – id UUID PK, first_name, last_name, dob, address, phone, email – encrypted where applicable.
Insurance table (insurances) – id UUID PK, provider_name, policy_number, group_number – encrypted where applicable.
Medical History table (medical_histories) – id UUID PK, diagnosis, treatment_notes, medications – encrypted where applicable.
User Role table (user_roles) – id UUID PK, role_name (admin, clinician, front_desk).
Audit Log table (audit_log) – id UUID PK, user_id UUID, action_type VARCHAR, resource VARCHAR, timestamp TIMESTAMP, details JSONB – includes entries for create, update, delete and now read actions.

### Patient Service
- **GET /api/v1/patients/{id}**
  - Auth: Bearer token with role claim (admin, clinician, front_desk).
  - Response: Patient resource JSON (PII fields encrypted at rest).
  - Errors: 401 (unauthenticated), 403 (unauthorized), 404 (not found), 429 (rate limit).
- **POST /api/v1/patients**
  - Creates new patient record; validates WCAG‑2.1 AA compliance (REQ‑001).
  - Errors: 400 (validation), 401, 403, 429.

### PDF Generation Service Integration (SVC‑001)
- **POST /api/v1/patients/{patient_id}/export/pdf**
  - Request schema:
    
    {
      "patient_id":"uuid",
      "requested_by":"string",
      "include_history":false
    }
    
  - Response schema:
    
    {
      "pdf_base64":"string",
      "watermark":"string",
      "generated_at":"date-time"
    }
    
  - Errors:
    - 400 Bad Request (missing/invalid fields)
    - 401 Unauthorized
    - 403 Forbidden (role lacks export_pdf permission)
    - 404 Patient not found
    - 409 Conflict (concurrent export limit reached)
    - 500 Internal Server Error (ERR‑006: Encryption failure during PDF generation)
    - 429 Too Many Requests (rate limit: 100 req/min per user, KPI‑042 target ≤100 ms latency)
  - Rate limiting is enforced by the API Gateway using token bucket algorithm; exceeding the limit returns 429 with `Retry-After` header.

### Error Codes
- **ERR‑001**: Transport TLS handshake failure.
- **ERR‑002**: Authentication token invalid or expired.
- **ERR‑003**: Authorization role mismatch.
- **ERR‑004**: Data validation error.
- **ERR‑005**: Service unavailable (downstream dependency failure).
- **ERR‑006**: Encryption/decryption operation failed (e.g., pgp_sym_encrypt error).
- **ERR‑007**: Audit log write failure.

## Data Models
- **Patient**: UUID primary key, encrypted columns (`first_name`, `last_name`, `ssn`, `dob`, `insurance_policy`). Non‑encrypted columns (`patient_id`, `created_at`).
- **AuditLog**: Immutable append‑only table with fields `operation`, `table_name`, `row_id`, `before_json`, `after_json`, `performed_by`, `created_at`. Includes read‑operation logging (`operation='SELECT'`). Partial index on `created_at` for retention policy (7 years) satisfying KPI‑003.

### Transport Encryption
- TLS 1.3 enforced on all external and internal HTTP traffic. Minimum cipher suite TLS_AES_256_GCM_SHA384. Mutual TLS required for service‑to‑service calls; client certificates stored as Docker secrets and rotated every 90 days.

### Role‑Based Access Control (RBAC)
- PostgreSQL roles: `admin`, `clinician`, `front_desk`. Row‑level security policies restrict `front_desk` to view only non‑clinical fields via view `patient_contact_view`. Clinicians can read all fields but cannot delete. Admins have full DML rights.

### Audit Logging Enhancements
- Added logging for READ operations (`SELECT`) to satisfy compliance audit requirements. Each read entry records `user_id`, `role`, `table_name`, `row_id`, `timestamp`, and a SHA‑256 hash of the returned row snapshot.

### Monitoring & Incident Response
- Prometheus metrics for TLS certificate expiry and key rotation timestamps. Alerts via Alertmanager when certificates <30 days to expiry or key rotation fails. Incident response procedures reference FR‑016.

## Rate Limiting Policy
- API Gateway enforces per‑user token bucket allowing 100 requests per minute. Exceeding limit returns HTTP 429 with `Retry-After` header. KPI‑042 target for request latency under rate limiting is ≤100 ms; alerts fire if average exceeds this threshold.

### POST /api/v1/intake/{id}/export-pdf
**Purpose**: Generate a PDF summary of the intake record with watermark.
**Request Body**: empty (uses path ID).
**Response**:
- 200 OK – `Content-Type: application/pdf` with PDF binary. Includes `X-Watermark-Timestamp` header.
- 400 Bad Request – if record not in exportable state (ERR-005).
- 401 Unauthorized – token missing/invalid (ERR-002).
- 403 Forbidden – RBAC violation (ERR-003).
- 404 Not Found – record missing (ERR-004).
- 503 Service Unavailable – Auth service down (ERR-006).
- 500 Internal Server Error – PDF generation failure after retries (ERR-005).
**Rate Limiting**: 100 requests per minute per user (KPI-043). Exceeding returns 429 with `Retry-After` header.

## Integration & Failure Handling

- **WeasyPrint Crash**: Service returns 500 with ERR-005 and retries up to two times before marking job failed.
- **PostgreSQL Audit Write Failure**: PDF generation proceeds; an `audit_entry_failed` event is emitted and response includes header `X-Audit-Warning: true`.
- **Auth Service Unavailable**: Endpoints return 503 with ERR-006 and include `Retry-After` header.
- **Read Audit Logging**: Every GET request creates an AuditLogEntry with action `read` to satisfy FR-003 compliance.

## Security & Compliance

- **Encryption**: All PHI fields encrypted using AES‑256‑GCM; key rotation governed by REQ‑007.
- **Authentication**: JWT signed with RS256 as defined in REQ‑007; token validation performed on each request.
- **Authorization**: Role‑Based Access Control enforced per FR‑002; clinicians can only view records of assigned patients.
- **Audit Logging**: All create, read, update, export actions logged per FR‑003 and retained for 7 years on immutable storage.
- **Rate Limiting**: Enforced at API gateway to meet KPI‑043; logs emitted for throttling events.
