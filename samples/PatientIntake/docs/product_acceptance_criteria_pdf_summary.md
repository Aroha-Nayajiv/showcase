# Acceptance Criteria for PDF Summary

### Personas
- PER-01: Clinician – medical professional who reviews patient intake data to make care decisions.
- PER-02: Front‑Desk Staff – administrative employee who enters patient information and provides receipt PDFs after submission.
- PER-03: Patient – individual providing personal health information via the web form and expecting confirmation of data capture.

### Acceptance Criteria
| AC ID | Story ID | Given | When | Then | Edge Cases 
|---|---|---|---|---|---|
| AC-001 | US-001 | Clinician is authenticated and has export permission (FR-002) | Clinician clicks "Export PDF" on a patient record | PDF generated with all required fields, includes watermark with clinician ID and ISO8601 timestamp, download starts within 2 seconds (KPI-030) | If patient record is locked for editing, export is blocked and error shown 
| AC-002 | US-001 | Clinician lacks export permission | Clinician attempts to export PDF | System displays "Access denied", no PDF generated, audit log entry recorded (FR-008) 
| AC-003 | US-002 | Front‑Desk staff has completed form submission and is authenticated | Staff clicks "Print Receipt" | PDF receipt includes patient name, submission timestamp, watermark "Generated by Front‑Desk" with staff ID, file size < 500 KB, generation completes within 1 second (KPI-030) | If network latency >200 ms, system falls back to server‑side generation and notifies user of delay 
| AC-004 | US-003 | Any authorized user opens an existing PDF export page | User opens PDF file | Watermark displays user ID and original export timestamp; if file older than 30 days, warning "Export older than 30 days – verify relevance" appears 
| AC-005 | US-001 | System audit logging enabled (AU‑6) | PDF export occurs | Immutable audit log entry records user_id, patient_id, action="pdf_export", timestamp, success flag; entry signed with HSM key and stored in append‑only storage (FR-008) 
| AC-006 | US-001 | Encryption at rest is active (FR-001) | PDF is generated and stored | PDF stored using field‑level encryption (pgcrypto); decryption only for authorized roles during download 
| AC-007 | US-004 | Admin accesses PDF export configuration UI | Admin updates role permissions 
| AC-008 | US-005 | Admin reviews exported PDFs for compliance

### Design Needs
- PDF generator must support dynamic text overlay for watermarks (e.g., WeasyPrint or weasyprint).
- Watermark format: "Exported by {UserID} at {ISO8601 timestamp}".
- PDF must be encrypted at rest using PostgreSQL pgcrypto field‑level encryption to satisfy FR‑001.
- Export operation must not exceed 2 seconds under normal load (KPI‑030).
- Audit log schema must include fields: user_id, patient_id, action="pdf_export", timestamp, success, checksum.
- Role‑based access control checks must reference permissions defined in FR‑002.

### Priority
- High (P1) – Required for HIPAA compliance and auditability (FR‑008, KPI‑030).
- Dependency: RBAC must be in place (FR‑002).

### Metrics
- Success rate of PDF export ≤ 2 seconds: ≥ 95 % of attempts.
- Watermark accuracy: 100 % of PDFs contain correct user ID and timestamp.
- Audit log completeness: 100 % of export actions logged immutably.

### User Stories

**US-001** *Clinician exports patient record as PDF*
- **Given** the clinician is authenticated and has the "export_pdf" permission
- **When** they request PDF export for a patient record
- **Then** the system generates a watermarked PDF within 2 seconds, logs the export event, and provides a download link that expires in 5 minutes.

**US-002** *Front‑Desk staff exports patient record as PDF*
- **Given** the staff member is authenticated with "export_pdf" permission
- **When** they select a patient and trigger export
- **Then** the same PDF is generated, logged, and a temporary download link is returned.

**US-003** *Administrator configures export permission*
- **Given** the admin accesses the access‑control settings page
- **When** they assign the "export_pdf" permission to roles Clinician and Front‑Desk
- **Then** users in those roles can export PDFs; others receive a 403 response.

**US-004** *Audit log entry creation*
- **Given** a PDF export has completed successfully
- **When** the system records the event
- **Then** an immutable audit log entry containing user_id, role, patient_id, action="export_pdf", timestamp, and PDF checksum is created within 500 ms.

**US-005** *Audit log verification tool*
- **Given** an admin views the audit log verification tool
- **When** they select a PDF export record
- **Then** the tool extracts watermark details (user ID, timestamp) from the PDF metadata and confirms they match the audit log entry; if extraction fails, the record is flagged for manual review.

### Non‑Functional Constraints
- PDF generation latency ≤ 2 seconds (95th percentile).
- Watermark opacity ≥ 30 %.
- Export event logged within 500 ms.
- PDFs stored encrypted using PostgreSQL pgcrypto.
- Download links are short‑lived signed URLs (expires 5 min).

### Compliance References
- HIPAA §164.312(a)(2)(iv) – Encryption of PHI in transit (TLS 1.3 termination inside Docker).
- HIPAA §164.312(b) – Audit controls (FR‑003 implementation).

### Endpoint Definitions

POST /api/v1/patients/{patient_id}/export-pdf
Headers: Authorization: Bearer <token>
Responses:
  200 OK – body: {"download_url":"https://..."}
  400 Bad Request – missing required fields
  401 Unauthorized – invalid or missing token
  403 Forbidden – insufficient permission
  504 Gateway Timeout – export processing exceeded time limit

GET /api/v1/audit/logs/{log_id}/verify-pdf
Headers: Authorization: Bearer <token>
Responses:
  200 OK – body: {"match":true}
  422 Unprocessable Entity – watermark extraction failed, flag for review

### Error Handling & Edge Cases
- Missing required patient fields → return 400 with error message, no PDF generated.
- Session timeout during export → return 401 prompting re‑authentication.
- Permission revoked mid‑export → current export completes; subsequent attempts return 403.
- Logging service outage → retry up to three times; on persistent failure, include warning in response but still deliver PDF.
- Corrupted PDF during verification → return 422 and flag record for manual audit.
