# Stakeholder Analysis Scope Definition

### 1. Overall System Boundary
- In‑scope: web‑based patient intake form, field‑level encryption at rest and in transit, PostgreSQL database with row‑level security, immutable audit log retained 7 years, PDF summary generation with watermark and timestamp, role‑based access control for admin/clinician/front‑desk, automated unit and integration tests, Docker‑Compose deployment in an air‑gapped environment.
- Out‑of‑scope: third‑party cloud services, mobile native applications, HL7/FHIR integration, advanced analytics dashboards.

### 3. Architecture Vision
The solution uses an open‑source stack: vanilla HTML/CSS/JavaScript front‑end, Python Flask back‑end, PostgreSQL 15 for persistence, and HashiCorp Vault for key management. TLS 1.3 secures all network traffic; PHI fields are encrypted with AES‑256 at rest. RBAC is enforced via PostgreSQL row‑level security policies. Audit logs are written to append‑only WORM storage with digital signatures. PDF summaries are generated server‑side with wkhtmltopdf and include a visible watermark containing the exporting user ID and timestamp plus hidden metadata for forensic verification. All components are containerised and orchestrated with Docker‑Compose, requiring no external network access after initial image pull.

### 1. Secure Data Capture (FR-001)
All patient demographic, insurance, and medical history fields must be encrypted in transit using TLS 1.3 and at rest with AES‑256. **Acceptance Criteria:** Network traffic analysis shows 100 % TLS 1.3 usage; storage encryption validated by OpenSSL checksum matching expected ciphertext.
**Test Cases:**
1. Intercept form submission with a proxy and verify TLS 1.3 handshake.
2. Retrieve stored PHI from the database and confirm fields are encrypted and match encryption key version.

### 2. Role‑Based Access Control (FR-002)
System enforces three RBAC tiers – Admin, Clinician, Front‑Desk – with least‑privilege principles. **Acceptance Criteria:** Access matrix tests confirm no role can read/write outside its tier; every access attempt is recorded in the audit log.
**Test Cases:**
1. Attempt to read a patient record as Front‑Desk staff and verify denial logged.
2. Perform write operation as Clinician and verify success and correct log entry.

### 3. Comprehensive Audit Logging (FR-003)
Every read and write operation generates an immutable log entry containing user ID, timestamp, operation type, and record ID; logs retained for 7 years on WORM storage. **Acceptance Criteria:** Log integrity verified by SHA‑256 hash chain; retention policy audited quarterly.
**Test Cases:**
1. Execute a series of CRUD operations and run reconciliation script to ensure log count matches operation count.
2. Attempt to modify an existing log entry and verify failure due to immutability.

### 4. PDF Summary Export with Watermark (FR-004)
Authorized staff may export a patient‑specific PDF summary. Each PDF must embed a visible watermark with export timestamp and exporting user ID and include hidden metadata for forensic verification. **Acceptance Criteria:** Sample PDFs inspected for watermark presence; metadata extracted matches audit log entry.
**Test Cases:**
1. Export a PDF as Clinician and verify watermark text includes correct user ID and timestamp.
2. Extract hidden metadata from the PDF and compare against corresponding audit log record.

### 5. Open‑Source Stack Constraint (FR-005)
All components must be built from OSI‑approved open‑source software (e.g., PostgreSQL, Flask, wkhtmltopdf). **Acceptance Criteria:** Dependency list generated by `pip freeze` contains only OSI‑approved licenses.
**Test Cases:**
1. Run license audit script and confirm 100 % compliance.

### 6. Air‑Gap Deployment (FR-006)
Docker‑Compose environment must operate without external network access after initial image pull. **Acceptance Criteria:** Runtime network scan shows zero outbound connections; deployment scripts include air‑gap verification step.
**Test Cases:**
1. Deploy containers on an isolated host and run `netstat` to confirm no outbound traffic.
2. Verify deployment completes successfully without pulling from external registries.

## Business Vision
The PatientIntake project delivers a HIPAA‑compliant, open‑source patient intake system that captures demographics, insurance, and medical history via a secure web form, stores data in an encrypted PostgreSQL database with role‑based access control, provides audited read/write logs, and generates PDF summaries with watermark and timestamp. The solution must be deployable on‑premises using Docker Compose without external network dependencies.

## Risks and Mitigations
| Risk ID | Description | Likelihood/Impact | Mitigation Action 
|---|---|---|---|
| RISK-001 | Unauthorized PHI access due to misconfigured RBAC | Medium/High | Automated role‑permission validation script run in CI; quarterly audit of access logs.
| RISK-002 | Encryption key compromise | Medium/High | Use HSM for key storage; rotate keys every 90 days via Vault; enforce MFA for key access.
| RISK-003 | Audit log tampering | Low/High | Write logs to append‑only immutable storage with digital signatures; retain backups for 7 years.
| RISK-004 | Deployment in non‑air‑gapped environment inadvertently exposing PHI | Low/High | Pre‑deployment script checks for external network interfaces and DNS resolution; abort if found.
| RISK-005 | Performance degradation under peak load affecting SLA | Medium/Medium | Conduct stress testing at 2× expected load; auto‑scale container resources; monitor response time KPI.

## Deployment Constraints
1. Docker Compose version ≥ 3.8 with healthchecks.
2. All images sourced from internal registry; no external pulls.
3. Network isolation enforced; containers communicate only over internal bridge.
4. Secrets stored encrypted in Git‑crypt; decrypted on target host.
5. Air‑gap verification script checks DNS resolution, public IP reachability, and Docker daemon binding.
6. Immutable audit log records each `docker compose up/down` action with user ID, timestamp, and SHA‑256 of compose file.
