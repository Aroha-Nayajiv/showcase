# PDF Summary Feature Specification

## PDF Summary Feature Specification: Primary User Personas

### 2. Front‑Desk Staff (Role ID: ROLE-FRONTDESK)
**Primary Goal:** Initiate PDF generation for a patient after completing the intake form, then provide the document to the patient or clinician as required.
**Key Needs:**
- Simple UI button labeled "Generate Intake Summary".
- Immediate visual confirmation that the PDF was generated (within 2 seconds, KPI-001).
- Automatic inclusion of a watermark showing staff ID, export time, and patient identifier (FR-008).
- Assurance that the action is logged (FR-003) and that only authorized roles can perform the export.
**Compliance Drivers:** Enforce RBAC so front‑desk staff cannot view PDFs for patients they are not assigned to (FR-002).
**Success Metrics:** Export latency ≤ 2 seconds for 95 % of attempts; error rate < 1 % for failed watermark insertion.

### 3. Patient (Role ID: ROLE-PATIENT)
**Primary Goal:** Receive a copy of their own intake summary for personal records, if permitted by policy.
**Key Needs:**
- Ability to request a PDF via a secure portal link after form submission.
- Watermark indicating "Patient‑Generated" with timestamp and patient ID.
- No exposure of other patients’ data; system must enforce that only the owning patient can request their own summary.
**Compliance Drivers:** HIPAA privacy rule; audit log must capture patient‑initiated export events (FR-003) and enforce consent tracking (FR-006).
**Success Metrics:** 100 % of patient‑requested PDFs include correct watermark; audit log entry created for every request.

### Cross‑Role Considerations
All roles share the requirement that every PDF export creates an immutable audit record (FR-003) and includes a cryptographic hash for integrity verification. Watermark format must be consistent: `Generated by {ROLE_ID} on {ISO8601_TIMESTAMP} – DocID:{UUID}`. The PDF generation service must run within the Docker Compose environment (FR-009) and use only open‑source libraries such as wkhtmltopdf or PDFBox.

### US-002 – Clinician
**Given** a clinician with role `clinician` is authenticated and has read permission on the patient record,
**When** the clinician selects "Export PDF Summary",
**Then** the system generates a PDF containing all submitted data, embeds a watermark `Generated by ROLE‑CLINICIAN on {timestamp} – DocID:{UUID}`, logs the export action with user ID and timestamp, and initiates the download within 2 seconds (KPI-001).

**Error Handling:**
- If the patient record is incomplete, an error dialog is shown and no PDF is generated.
- If the clinician lacks permission, an "Access Denied" error is shown and an unauthorized_export audit entry is recorded.
- If watermark insertion fails, the PDF is still generated but includes a placeholder "Watermark unavailable" notice; an error audit entry records the failure.

## Design Needs
- **PDF Generation Library:** Use wkhtmltopdf (HTML → PDF) or Apache PDFBox for direct PDF creation; both support programmatic watermark overlay.
- **Watermark Implementation:** Overlay text layer containing role ID, ISO‑8601 timestamp, and UUID; fallback placeholder on failure.
- **Encryption Key Management:** Keys stored in HashiCorp Vault; rotation every 90 days; access controlled via IAM policies.
- **Audit Log Schema:** Table `audit_log` with columns `event_id`, `event_type`, `user_id`, `role_id`, `timestamp`, `details_json`. Export events recorded with `event_type='export_pdf'`.
- **RBAC Matrix:** Define permissions in table `role_permissions` linking roles to actions (`export_pdf`, `configure_export`).

All user stories trace to FR‑001, FR‑002, FR‑003 and KPIs KPI‑001, KPI‑006.

### 1. Overview
The PDF Summary feature enables authorized staff to export a patient\u2013intake summary as a tamper\u2013evident PDF. It satisfies HIPAA security and audit requirements by applying role\u2013based access control, AES\u201326 encryption at rest, TLS\u00a01.3 transport, and immutable watermarks.

### 3. Detailed User Stories with Acceptance Criteria (BDD)
**US-001**

Given a front\u2011desk staff member has completed a patient intake form
When they select "Export PDF" for that patient
Then the system generates a PDF within 2\u00a0seconds
And the PDF contains a watermark with user ID and ISO\u20118601 timestamp
And an on\u2011screen confirmation "PDF generated successfully" is shown

**US-002**

Given a clinician is viewing a patient record
When they open the attached PDF summary
Then the PDF displays the same clinical data as the web view
And the watermark matches the logged export event (user ID, timestamp)
And any mismatch triggers a "Tamper\u2011evidence violation" alert

**US-003**

Given an administrator accesses the export settings page
When they assign export permission to role "clinician"
Then only users with that role can invoke the PDF export API
And attempts by unauthorized roles return HTTP\u00a0403 with error "Export not authorized"

**US-004**

Given a front\u2011desk staff member clicks "Export PDF"
When the generation completes
Then a toast notification "PDF generated in 1.8\u00a0s" appears
And the timestamp logged is ≤\u00a02\u00a0seconds from request start

**US-005**

Given an auditor selects "Export audit\u2011linked PDF" for a date range
When the system creates the PDF bundle
Then each record includes export metadata (user ID, patient ID, timestamp, watermark hash)
And any hash mismatch is flagged as "tamper\u2011evidence violation"

**US-006**

Given a patient logs into the portal and requests their intake summary
When they click "Download PDF"
Then the system generates the same tamper\u2011evident PDF as staff
And logs an export event with patient\u2011as\u2011user ID

### 4. API Contract
**Endpoint:** `POST /api/v1/patients/{patient_id}/export/pdf`
**Headers:** "Authorization: Bearer <token>"; "Content-Type: application/json"
**Request Body:** `{ "requester_id": "<user_id>", "purpose": "clinical_review" }`
**Responses:**
- `200 OK` – binary PDF stream; headers include `X-Watermark-Hash`.
- `403 Forbidden` – when requester lacks export permission.
- `429 Too Many Requests` – rate limit exceeded.
**Security:** TLS\u00a01.3 enforced; JWT signed with RSA\u201326; audit log entry created atomically.

### 5. Error Handling
| Scenario | Expected Behavior |
|---|---|
| Watermark generation failure | Return `500 Internal Server Error` with message "Watermark service unavailable"; log error and trigger alert email to security officer |
| Unauthorized export attempt | Return `403 Forbidden` with message "Export not authorized"; audit log records attempt and user ID |
| PDF generation exceeds 2\u00a0seconds | Return `202 Accepted` with async status URL; notify user of delay and log performance metric |
| Audit log write failure (immutability breach) | Abort operation, return `500`; create security alert; ensure original log entry remains unchanged |
