# PDF Summary Feature Specification

## PDF Summary Feature Specification

### Overview
The PDF Summary feature enables authorized staff to generate a tamper‑evident PDF intake summary for each patient. The MVP supports generation, watermarking with user ID and timestamp, secure storage, and audit logging that satisfy HIPAA technical safeguard requirements (45 CFR §164.312(a)(2)(iv)) and NIST SP 800‑53 controls AC‑2, AC‑3, AU‑6, AU‑12.

### User Stories
| ID | Persona | Goal | Benefit |
|----|---------|------|---------|
| US-001 | Clinician | generate a PDF summary of a patient's intake record | I can review the patient's information offline while maintaining HIPAA compliance |
| US-002 | Front Desk Staff | export a PDF summary for a patient after registration | I can provide the patient with a printable copy that includes a watermark and timestamp |
| US-003 | Administrator | audit PDF export activity | I can ensure that only authorized roles performed exports and that logs are tamper‑evident |
| US-004 | Front Desk Staff | receive an error if I attempt to export a PDF for a patient without proper role assignment | the system prevents unauthorized disclosure of PHI |
| US-005 | Clinician | receive a warning if PDF generation exceeds 2 seconds | performance remains within the SLA defined in KPI‑002 (p95 <200 ms) and does not degrade user experience |

### Acceptance Criteria
| AC ID | Story ID | Given | When | Then | Edge Cases |
|-------|----------|-------|------|------|------------|
| AC-001 | US-001 | the clinician is authenticated and has the "export_pdf" permission and a completed intake record exists | the clinician clicks "Generate PDF" for the record | a PDF is streamed over TLS 1.3, contains all required fields, embeds a watermark "Generated by {user_id} on {ISO8601_timestamp}", and is stored encrypted at rest; the action is logged with user ID, patient ID, timestamp, and outcome | if required fields are missing the system shows an error and does not generate a PDF |
| AC-002 | US-001 | the PDF generation library is available and the encryption key is loaded | the generation process exceeds 200 ms | the system still returns the PDF but logs a performance warning; 95 % of requests must complete ≤500 ms; if the library fails an error message is shown and the event is logged with severity "error" |
| AC-003 | US-002 | front‑desk staff is logged in with "export_pdf" role | the staff selects "Export PDF" for a newly submitted intake form | the PDF is generated with the same watermark rules, the export timestamp is recorded in audit log AU‑6, and the file is delivered via HTTPS to the staff's download folder | if the staff attempts to export a record belonging to another clinician access is denied and an "access denied" audit entry is created |
| AC-004 | US-003 | an administrator accesses the audit‑log UI | the admin filters logs for "PDF export" events in the last 24 hours | the UI displays a list of export events showing user ID, patient ID, timestamp, and success/failure status; the list can be exported as CSV for compliance review |
| AC-005 | US-001 | the system is configured with field‑level encryption at rest (AES‑256) and TLS‑1.3 in transit | a PDF is generated and stored on disk | the stored PDF file is encrypted at rest using the same key hierarchy; decryption occurs only for authorized downloads; if decryption fails due to key rotation the download is blocked and an alert is raised |
| AC-006 | US-001 | the generated PDF passes integrity check using SHA‑256 hash stored in audit log | a user attempts to open a modified PDF file | the system detects hash mismatch, blocks opening, and logs a tamper‑evidence event; if the hash service is unavailable a warning is displayed and the attempt is logged |

### API Endpoints
| Method | Path | Description | Required Role |
|--------|------|-------------|----------------|
| POST | /api/v1/pdf/export/{record_id} | Generate and stream PDF for given intake record | export_pdf |
| GET | /api/v1/pdf/audit?start={date}&end={date} | Retrieve audit entries for PDF exports within date range | admin |

### Error Handling Enhancements
- **Encryption key failure**: returns HTTP 500 with message "Encryption key unavailable" and creates an audit entry with severity "critical".
- **Performance timeout**: if generation exceeds 5 seconds the request is cancelled and HTTP 504 is returned; a performance alert is logged.
- **Missing permissions**: backend returns HTTP 403 and logs an "access denied" audit event.
- **Library unavailable**: returns HTTP 503 with retry‑after header and logs service‑unavailable event.

### Performance Monitoring
Instrumentation records start‑time and end‑time for each export. Alerts trigger when median latency >200 ms over a 5‑minute window. KPI‑002 (p95 <200 ms) is tracked in the monitoring dashboard.

### Audit Logging Details
Each export creates an immutable log entry containing:
- `event_type`: "pdf_export"
- `user_id`
- `patient_id`
- `record_id`
- `timestamp`
- `outcome`: "success" or error code
- `hash`: SHA‑256 of generated PDF
Logs are stored in append‑only storage with digital signatures to satisfy FR‑003 and KPI‑017.

### Security Controls Summary
- **Transport**: TLS 1.3 enforced via reverse proxy.
- **At‑rest encryption**: AES‑256 using per‑record keys rotated quarterly.
- **Access control**: RBAC checked against PostgreSQL row‑level security policies.
- **Tamper evidence**: SHA‑256 hash verification on download; mismatch triggers alert.

### 3. Design Needs

- Watermark Engine: Must be implemented using an open‑source PDF library (e.g., pdf-lib for Node.js or PyPDF2 for Python) that supports overlaying transparent text on each page.
- Timestamp Formatting: Use ISO‑8601 UTC by default; configurable per admin settings; ensure locale‑aware rendering for audit purposes.
- Security Controls: PDF generation runs in a sandboxed container with read‑only file system except for a dedicated `/tmp/pdf` volume; output files are encrypted at rest using AES‑256. Encryption key retrieval failures produce error 503.
- Audit Logging: Every export creates an audit log entry (FR-003) capturing user ID, patient ID, timestamp, file hash (SHA‑256), outcome status (success/failure), encryption key identifier, and any error codes.
- Performance Requirement: PDF generation must complete within 1.5 seconds for records under 5 KB of JSON payload (measured on reference hardware Intel i5, 8 GB RAM) to satisfy KPI‑002 (performance).
- Compliance Checks: Include automated verification that each PDF contains the required watermark and timestamp; CI pipeline runs a script that parses generated PDFs and fails the build if missing.
- API Definition: POST `/api/patient/{patient_record_id}/export-pdf` with JSON body `{ "initiator": "user_id", "timestamp": "ISO8601", "options": { "watermark": "string", "format": "PDF" } }`. Returns `200 OK` with `application/pdf` payload or appropriate error codes.
