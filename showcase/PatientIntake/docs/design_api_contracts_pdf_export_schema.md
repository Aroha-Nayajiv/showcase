# Patient Intake System – Technical Design (Refined)

### 5. Overview
This document defines the PostgreSQL relational schema, role‑based access control (RBAC) policies, and supporting indexes that enable tamper‑evident logging while preserving performance for a typical departmental SaaS deployment (≤10 000 concurrent users). All tables leverage PostgreSQL 15 features such as row‑level security (RLS) and native `JSONB` columns for extensibility.

### 6. Table Definitions

#### patients
sql
CREATE TABLE patients (
    patient_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    first_name TEXT NOT NULL,
    last_name TEXT NOT NULL,
    date_of_birth DATE NOT NULL,
    ssn BYTEA NOT NULL,               -- pgp_sym_encrypt(ssn)
    address BYTEA NOT NULL,            -- encrypted address
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

#### insurances
sql
CREATE TABLE insurances (
    insurance_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    patient_id UUID NOT NULL REFERENCES patients(patient_id) ON DELETE RESTRICT,
    carrier_name TEXT NOT NULL,
    policy_number BYTEA NOT NULL,
    group_number BYTEA,
    effective_date DATE,
    expiration_date DATE,
    created_at TIMESTAMPTZ DEFAULT now()
);

#### medical_histories
sql
CREATE TABLE medical_histories (
    history_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    patient_id UUID NOT NULL REFERENCES patients(patient_id) ON DELETE RESTRICT,
    history_data JSONB NOT NULL   -- encrypted fields inside JSONB
);

#### audit_logs
sql
CREATE TABLE audit_logs (
    audit_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    actor_role TEXT NOT NULL CHECK (actor_role IN ('admin','clinician','front_desk')),
    actor_id UUID NOT NULL,
    action TEXT NOT NULL CHECK (action IN ('CREATE','READ','UPDATE','DELETE')),
    entity TEXT NOT NULL CHECK (entity IN ('patients','insurances','medical_histories')),
    entity_id UUID NOT NULL,
    timestamp TIMESTAMPTZ DEFAULT now(),
    outcome TEXT NOT NULL CHECK (outcome IN ('SUCCESS','FAILURE')),
    details BYTEA NOT NULL,               -- encrypted audit details
    prev_hash BYTEA NOT NULL                -- SHA‑256 hash chain value
);

### 7. Indexes & Performance Optimizations
- **B‑tree index** on `audit_logs(timestamp)` – fast range queries for compliance reports.
- **GIN index** on `medical_histories.history_data` – efficient JSONB search.
- **Composite index** `(actor_role, action)` – accelerates audit dashboard queries.
- **Partial index** on `audit_logs(outcome)` for quick failure‑rate analysis.

### 8. Retention & Archival Strategy
- Audit records retained **minimum 7 years** per HIPAA 164.308(a)(1)(ii)(C).
- Nightly `pg_dump` of `audit_logs` partitioned by month → encrypted off‑site object store.
- Live table retains only the most recent **12 months** for operational queries.
- Archival partitions are read‑only and indexed for historical investigations.

## PDF Export Service Interface (SVC‑001)

### 10. Service Boundary
| Service | Responsibility | Dependencies | Produced Event |
|---------|------------------|--------------|----------------|
| SVC‑001 | Generate PDF intake summary, apply watermark & timestamp, enforce RBAC | PostgreSQL (`patients`, `audit_logs`), wkhtmltopdf PDF library, AES‑256‑GCM encryption module, Auth service (JWT validation) | `PDFExported` (payload includes `patient_id`, `file_id`, `timestamp`) |

### 11. API Contract

#### 11.1 Endpoint Table
| Method | Path | Description | Query Parameters | Success Response | Security |
|--------|------|-------------|-------------------|------------------|----------|
| GET | `/api/v1/patients/{patient_id}/export/pdf` | Return a newly generated PDF summary for the specified patient | `include_watermark` (bool), `timestamp_format` (string) | `200 OK` with body defined in **Response Schema** | Bearer Token |

#### 11.2 Response Schema (`SCH-PDF-RESP-001`)

- **pdf_url**: https://files.internal/pdfs/123e4567-e89b-12d3-a456-426614174000.pdf
- **expires_at**: 2026-06-05T15:30:00Z
- **request_id**: 123e4567-e89b-12d3-a456-426614174000

*`pdf_url` is a pre‑signed URL valid until `expires_at`. The URL points to a read‑only location inside the air‑gapped Docker network.*

#### 11.3 Error Handling Specification (`SCH-PDF-ERR-001`)
| HTTP Code | Error Code | Description | When Returned |
|-----------|------------|-------------|----------------|
| 400 | `INVALID_REQUEST` | Malformed request payload or missing required query parameters. | Validation failure on input values |
| 401 | `UNAUTHORIZED` | Missing or invalid JWT token. | Authentication failure |
| 403 | `FORBIDDEN_ROLE` | Authenticated token does not contain an allowed role (`admin`, `clinician`, `front_desk`). | Authorization check fails |
| 404 | `PATIENT_NOT_FOUND` | No patient record matches the supplied `patient_id`. | Resource lookup fails |
| 429 | `RATE_LIMIT_EXCEEDED` | Caller exceeded the per‑minute request quota (default = 60 req/min). | Rate limiting enforcement |
| 500 | `INTERNAL_ERROR` | Unexpected server error during PDF generation or storage. | Unhandled exception |

All error responses follow the same JSON envelope:

- **error_code**: ...
- **message**: Human readable description
- **request_id**: ...

#### 11.4 Rate Limiting & Monitoring
- **Rate limit**: 60 requests per minute per service account; exceeded calls receive HTTP 429.
- **Metrics** exposed via Prometheus endpoint `/metrics`: `pdf_export_requests_total`, `pdf_export_success_total`, `pdf_export_error_total{code}`.
- **Alerting**: Trigger on error rate > 2% over a rolling 5‑minute window.

### 12. Security Considerations
1. **Authentication & Authorization** – Every request must contain a valid JWT (`alg: RS256`) issued by the Auth service. The token must include the claim `role`; the service checks that the role is one of `admin`, `clinician`, or `front_desk`. Access control follows the matrix defined in **FR‑001**.
2. **Transport Encryption** – All traffic uses TLS 1.3 (`tls_version` defined in project globals). No plain‑text URLs are ever returned.
3. **At‑Rest Encryption** – Generated PDFs are stored encrypted with AES‑256‑GCM; the encryption key is master keys rotated every 90 days, per-field DEKs every 30 days and referenced via `app.pdf_key_version`.
4. **Tamper Evidence** – Each successful export creates an entry in `audit_logs` with action=`READ`, entity=`patients`, outcome=`SUCCESS`, and includes the generated `request_id` for traceability.
5. **Audit Correlation** – The downstream service that consumes the `PDFExported` event must include the same `request_id` to allow end‑to‑end verification of compliance with **KPI-003**.

## 14. System Architecture & Integration Patterns

| Component | Responsibility | Communication Protocol |
|------------|------------------|--------------------------|
| **PDF Export Service** (`pdf-export-service`) | Generates PDFs from patient data, applies encryption and watermarking | REST over internal network (`http://pdf-export-service:8080`) |
| **Audit Service** (`audit-service`) | Persists audit events (`PDFExported`) with full payload | REST (`http://audit-service:8080`) |
| **Key Management Service (KMS)** | Supplies AES‑256‑GCM keys; keys rotated weekly via endpoint defined in project globals (`KMS_ENDPOINT`) | HTTP API |
| **PostgreSQL DB** (`patient_intake`) | Stores patient demographics and clinical data used for PDF generation | Direct DB connection (read‑only for export) |
| **Internal Network (`internal_net`)** | Isolates services to satisfy air‑gap requirement | Docker Compose network |

The services are deployed as independent containers orchestrated by Docker Compose (see Section 6). All inter‑service traffic stays within `internal_net`, satisfying the air‑gap and multi‑tenant isolation requirements.

### 15.2 Audit Event Payload (sent to Audit Service)

{
  "event_type": "PDFExported",
  "event_timestamp": "2026-05-05T12:35:01Z",
  "patient_id": "string",
  "user_id": "string",
  "action": "export_pdf",
  "outcome": "success" | "failure",
  "export_id": "string",
  "pdf_sha256": "string",
  "error_code": null | "string"
}

The Audit Service stores the event in its PostgreSQL database and forwards it to the centralized compliance pipeline.

### 15.3 Encryption & Key Rotation
* **Algorithm**: AES‑256‑GCM (authenticated encryption).
* **Key Source**: Retrieved from KMS via `KMS_ENDPOINT` defined in project globals.
* **Rotation Policy**: Keys are rotated weekly by the KMS; the service fetches the latest key on each export request.
* **Key Handling**: Keys are never persisted on disk; they reside only in memory for the duration of the encryption operation.

### 15.4 Watermark Enforcement
The watermark text is constructed as:

"Exported by {user_id} at {UTC_timestamp}"

It is embedded into the PDF using the PDF library (`wkhtmltopdf`). The watermark is part of the encrypted payload, guaranteeing non‑repudiation.

### 15.5 Audit Logging Guarantees
* Every export attempt—successful or failed—produces an audit event.
* The payload includes a SHA‑256 hash of the final PDF to enable integrity verification.
* Audit events are transmitted over TLS 1.3 within `internal_net`.
