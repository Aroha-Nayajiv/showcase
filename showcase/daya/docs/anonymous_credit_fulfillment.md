# Anonymous Credit Fulfillment & Absolute Anonymization

## 1. Virtual Token Presentation & Merchant Visibility

The core product mechanism for PII isolation is the single-use virtual card token. The Beneficiary presents this token at the POS, and the Merchant system must interact with it as a standard payment instrument.

### Acceptance Criteria

**AC-1.1.1: Token Visual Fidelity**
The virtual card token displayed in the Beneficiary app (or Apple/Google Wallet pass) must be visually indistinguishable from a standard consumer gift card or credit card. It must display a masked card number (e.g., `•••• •••• •••• 1234`), an expiration date, and a CVV. No branding, logos, or text indicating "Daya," "MealCredit," "Donation," or "Charity" may appear on the token itself.

**AC-1.1.2: Zero PII at POS Interface**
The Merchant POS interface (or zero-footprint integration dashboard) must only receive the token value, the transaction amount, and the Merchant Category Code (MCC) for validation. The POS must not display, log, or transmit any beneficiary name, ID, or demographic data. The transaction must appear to the Merchant as a standard anonymous card-present transaction.

**AC-1.1.3: MCC & Location Locking**
The token must be cryptographically locked to the specific MCC (e.g., 5812 - Eating Places) and the specific restaurant location ID. If the Beneficiary attempts to use the token at a non-participating MCC (e.g., a liquor store) or a non-participating location, the transaction must be declined at the Stripe network layer before reaching the Merchant POS, with a generic "Declined" message shown to the Beneficiary.

## 2. Absolute Anonymization Enforcement

This section details the product rules that ensure no PII crosses the platform boundary during the redemption flow, specifically addressing the financial_clearing surface's role in stripping PII before it reaches the Merchant POS.

### Product Rules for PII Isolation

**PR-2.1: Financial Clearing Surface PII Stripping**
The financial_clearing surface acts as the mandatory PII stripping boundary. Before any transaction data is forwarded to the Merchant POS or the MerchantFulfillment payout ledger, the financial_clearing surface must:
1. Remove all Beneficiary PII (names, contact details, demographic flags) from the transaction payload.
2. Remove all Donor PII from the transaction payload.
3. Retain only the transaction ID, the anonymized token reference, the transaction amount, the timestamp, and the Merchant location ID.

**PR-2.2: Immutable Receipt Generation**
Donors must receive immutable transactional receipts within 120 seconds of redemption. These receipts must strictly prohibit the transmission of any identifying beneficiary parameters. The receipt must confirm the impact (e.g., "Your donation funded a meal") without revealing who consumed it.

**PR-2.3: Production Log Sanitization**
No PII such as legal name or domestic background may be stored on-platform or in production logs. Any logs generated by the financial_clearing surface must be automatically sanitized to ensure that no Beneficiary or Donor identity can be reconstructed from log data.

## 3. Offline Operational Resilience

To support Offline Operational Resilience (CON-10A76F2185), the product must define how the Merchant POS handles token validation when connectivity is lost. This section focuses on the product-level requirements for resilience, avoiding premature technical implementation details.

### Product Requirements for Offline Resilience

**PR-3.1: Offline Transaction Handling**
The product must allow Merchants to process transactions when the POS system is offline. The specific mechanism for offline validation (e.g., cached token lists, cryptographic offline signatures) is a technical design decision. The product requirement is that the system must ensure no double-spending of credits occurs, even in offline scenarios.

**PR-3.2: Reconciliation Protocol**
When connectivity is restored, the POS system must automatically reconcile offline transactions with the central ledger. The product must define the success criteria for reconciliation (e.g., all offline transactions must be reconciled within 24 hours) but leave the specific synchronization protocol to the technical design phase.

**PR-3.3: Merchant Throttle Parameters**
Kitchens must be able to toggle real-time throttle parameters to prevent structural overload. The product must expose a configurable limit for the maximum number of MealCredit orders per hour that a Merchant can process. This limit is configurable by the Merchant Administrator.

## 6. Knowledge Gaps & Assumptions

**ASSUMPTION:** The Stripe Issuing API supports the generation of single-use, MCC-locked virtual cards with the required cryptographic signature for offline validation. If Stripe does not support this natively, a custom tokenization layer may be required, which is currently out of scope for this product spec.

**KNOWLEDGE_GAP:** The exact mechanism for "zero-footprint" POS integration is not defined. This artifact assumes that Merchants can ingest the token via a standard payment gateway API or a lightweight edge dashboard. The specific UI/UX for this dashboard is owned by the MerchantFulfillment journey artifact.

**KNOWLEDGE_GAP:** The specific "ineligible purchase" filtering rules (e.g., alcohol, non-food) are defined at the Stripe network layer. The product spec assumes these rules are pre-configured in the Stripe Issuing environment and does not define the logic for them here.

**KNOWLEDGE_GAP:** The exact data contract for anonymization (e.g., specific fields to be hashed, salted, or dropped) is not defined. This artifact assumes that PII is stripped at the point of entry into the financial_clearing surface. The specific data sanitization rules must be established by the security/compliance artifact.

**KNOWLEDGE_GAP:** The exact threshold for Offline Success Rate (SM-1.4.5) is not established in project truth. This metric requires a decision from the Product/Design team to define an acceptable reconciliation success rate.
