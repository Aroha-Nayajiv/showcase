# Frontend / Presentation Layer

## 1. Offline-First State Management and Secure Token Architecture

This section defines the local data persistence, state synchronization, and cryptographic token storage mechanisms for the React Native client (Expo v51 / Fabric). It ensures that the Recipient (ACT-DC00FA84DC) and Contributor (ACT-2A20B038B1) actors can operate securely in low-connectivity environments while maintaining strict PCI-DSS Level 1 (CON-6EA64CF2A1) and SOC2 Type II (CON-0A6423E6B0) compliance.

### 1.1. Secure Offline Token Storage (CON-0346AE051D)

To prevent token forgery in offline mode, the application must never store raw financial credentials or sensitive PII in AsyncStorage or plain-text local state. Instead, we utilize the device's hardware-backed Secure Enclave (iOS) or Keystore (Android) via `react-native-keychain`.

**Architecture:**
*   **Storage Mechanism:** `react-native-keychain` is used to store the SecureToken (a single-use virtual card token generated by the Core Transaction & Ledger Service (SUR-DD602DB92C)).
*   **Cryptographic Binding:** The token is encrypted using a key derived from the device's biometric authentication (FaceID/TouchID) or passcode. This ensures that even if the device is compromised, the token cannot be extracted without biometric verification.
*   **Data Isolation:** No raw credit card data or beneficiary PII is ever persisted locally. The AsyncStorage is used exclusively for non-sensitive UI state (e.g., last known location, cached merchant list IDs) and the OfflineQueue metadata.

**Token Lifecycle:**
1.  **Generation:** The Contributor (ACT-2A20B038B1) or Recipient (ACT-DC00FA84DC) requests a token via the GraphQL API. The backend generates a single-use, pseudo-anonymized token.
2.  **Storage:** The token is immediately stored in the Secure Enclave. The UI displays a QR code or barcode representation.
3.  **Offline Use:** If the device is offline, the QR code is generated from the locally stored token. The token includes a cryptographic nonce to prevent replay attacks.
4.  **Synchronization:** When connectivity is restored, the OfflineQueue flushes any pending transactions. The backend validates the token's signature and nonce.

*   `KNOWLEDGE_GAP: SecureToken - The exact cryptographic algorithm and key derivation function (KDF) for the device-level SecureStore must be established by the Security Architecture & Access Control artifact. Evidence needed: PCI-DSS Level 1 requirements for key management in mobile environments.`

### 1.2. Offline Fallback QR/Barcode Presentation (CON-036979982A)

To ensure compatibility with standard, low-tech POS devices, the offline fallback presentation must use universally scannable formats.

**Presentation Contract:**
*   **Format:** QR Code (ISO/IEC 18004) or EAN-13 Barcode. QR Code is preferred for its higher data capacity and error correction.
*   **Data Payload:** The payload must contain the SecureToken and a short-lived cryptographic nonce (TTL: 5 minutes). This prevents replay attacks if the QR code is photographed and reused.
*   **Visual Design:** The QR code must be high-contrast and large enough to be scanned by standard POS scanners from a distance of 10-15 cm. It must be visually identical to a consumer gift card to reduce social stigma.
*   **Accessibility:** The QR code must be accompanied by an audio description for low-vision beneficiaries, adhering to WCAG 2.1 AA standards (CON-ADA1B3458C).

**Offline Generation Logic:**
1.  **Token Retrieval:** The app retrieves the SecureToken from the Secure Enclave.
2.  **Nonce Generation:** A new nonce is generated locally using a cryptographically secure random number generator.
3.  **Encoding:** The token and nonce are encoded into a QR code string.
4.  **Rendering:** The QR code is rendered using a high-performance canvas component to ensure smooth performance on Fabric architecture.

### 1.3. Offline Queue and State Synchronization

The OfflineQueue is a persistent, append-only log of transactions that occurred while the device was offline. It ensures that no financial data is lost and that the local state eventually converges with the server state.

**Queue Structure:**
*   **Storage:** `AsyncStorage` (encrypted) or a local SQLite database (via `react-native-sqlite-storage`) for high-volume transaction logs.
*   **Entry Schema:** Each entry contains a unique `transaction_id`, `timestamp`, `action_type` (e.g., REDEMPTION, DONATION), `payload` (the transaction details), and `status` (PENDING, COMPLETED, FAILED).
*   **Conflict Resolution:** In the event of a conflict (e.g., double-spending), the server's timestamp takes precedence. The client must implement a "last-write-wins" policy for non-financial state and "first-come-first-served" for financial state.

**Synchronization Logic:**
1.  **Network Detection:** The app uses `NetInfo` to detect network connectivity changes.
2.  **Queue Flush:** When connectivity is restored, the OfflineQueue is flushed to the API Gateway & Orchestration Layer (SUR-D6FFF7036F).
3.  **Retry Logic:** Failed transactions are retried with exponential backoff. After 3 failed attempts, the transaction is marked as FAILED and the user is notified.
4.  **State Reconciliation:** The client receives a delta of changes from the server and updates the local state accordingly.

### 1.5. Error Handling and Recovery Flows

This section defines the error handling strategy for the Client Application Layer, ensuring that failures in the Contributor Primary Transaction Flow (JNY-4FC1874968) and Recipient Discovery & Redemption (JNY-76281D3F3C) are handled gracefully without exposing internal system states or raw PII.

**Error Classification:**
Errors are classified into three categories to determine the appropriate user-facing response and recovery mechanism:

| Error Category | Description | User-Facing Action | System Action |
| :--- | :--- | :--- | :--- |
| **Transient Network Error** | Temporary loss of connectivity or server timeout. | Display "Connection Lost" banner; enable offline mode if available. | Queue request in OfflineQueue; retry with exponential backoff. |
| **Authentication Error** | Invalid or expired session token. | Prompt user to re-authenticate. | Clear local session state; redirect to login screen. |
| **Financial Validation Error** | Transaction rejected by backend (e.g., insufficient funds, invalid token). | Display specific error message (e.g., "Insufficient Credits"). | Log error to CloudTrail; mark transaction as FAILED in OfflineQueue. |

**Stripe Issuing Partial Failure Handling:**
When Stripe Issuing returns a partial failure (e.g., authorization pending, MCC mismatch), the client must:
1.  Parse the Stripe error code (e.g., `card_declined`, `insufficient_funds`).
2.  Map the error code to a user-friendly message.
3.  If the error is due to an MCC mismatch, inform the user that the selected merchant does not accept MealCredits.
4.  Log the error event with the Stripe error code for backend reconciliation.

**Knowledge Gap:**
*   `KNOWLEDGE_GAP: Specific Stripe Issuing error codes and their corresponding user-friendly messages must be defined. Evidence needed: Stripe Issuing API documentation for error handling best practices.`

### 1.6. Accessibility and WCAG 2.1 AA Compliance (CON-ADA1B3458C)

To ensure low-vision beneficiaries can use the platform effectively, the Client Application Layer must adhere to WCAG 2.1 AA standards.

**Screen Reader Strategy:**
*   **Semantic Labels:** All interactive elements (buttons, inputs, cards) must have descriptive `accessibilityLabel` props.
*   **Focus Management:** Focus must be programmatically managed during route transitions and modal presentations.
*   **Dynamic Type:** The UI must support dynamic text sizing up to 200% without breaking layout.

**Offline Fallback Presentation Contract:**
*   **QR Code Accessibility:** The QR code component must include an `accessibilityLabel` that describes the token value (e.g., "MealCredit Token for $10.00 at [Merchant Name]").
*   **Audio Feedback:** Successful token generation and redemption must trigger a haptic or audio cue.

**Knowledge Gap:**
*   `KNOWLEDGE_GAP: Specific WCAG 2.1 AA success criteria for QR code accessibility must be defined. Evidence needed: WCAG 2.1 guidelines for non-text content.`
