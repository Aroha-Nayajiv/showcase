# PII Segregation & Anonymization Strategy

This section defines the cryptographic hashing schema and data isolation controls for the MealCredit platform, ensuring strict adherence to [CON-0A0288EED4](../project_glossary.md#con-0a0288eed4) (strict data isolation), [CON-92F07E31B0](../project_glossary.md#con-92f07e31b0) (cryptographic segregation), and [CON-FCFF86A326](../project_glossary.md#con-fcff86a326) (cryptographic hashing layers). The objective is to enable [CON-23A501C051](../project_glossary.md#con-23a501c051) and [CON-413928CB1C](../project_glossary.md#con-413928cb1c) (anonymous correlation of donor impact to beneficiary redemption) while ensuring zero raw PII (legal names, demographic status) is stored in the Aurora PostgreSQL ledger ([SUR-FA61592CD4](../project_glossary.md#sur-fa61592cd4)).

### 1.1 Cryptographic Hashing Schema

To satisfy the requirement for absolute anonymization, all beneficiary PII must be transformed into irreversible hashes before being written to the financial ledger or redemption event tables. The hashing mechanism must be deterministic to allow for consistent lookups in the mapping table, but non-reversible to prevent PII extraction.

Algorithm: SHA-256. This provides a 256-bit hash output, which is cryptographically secure against pre-image attacks for the scale of 50,000 MAU.
Salt Strategy: A project-specific, environment-rotating salt is required. The salt must be stored in AWS Secrets Manager, not in the database, to prevent rainbow table attacks if the database is compromised.
Salt Versioning: Each hash must be associated with a salt_version (UUIDv4) to allow for salt rotation without re-hashing all historical records immediately. This ensures that old hashes can be validated against previous salts during a transition period.

### 1.2 Data Isolation Architecture

The Aurora PostgreSQL database (SUR-FA61592CD4) must be logically segmented to enforce strict data isolation. Raw PII and financial data must never coexist in the same queryable surface.

PII Storage: Raw PII (legal names, demographic status) is stored in a dedicated, highly restricted schema (e.g., pii_raw). Access to this schema is restricted to the Identity & Access Management ([CAP-IDENTITY-ACCESS-MANAGEMENT](../project_glossary.md#cap-identity-access-management)) service only.
Financial Ledger: The financial ledger and redemption event tables will ONLY store the hash_value and salt_version. No PII fields are permitted in these tables.
Mapping Table: A dedicated pii_analytics_mapping table serves as the ONLY link between a PII hash and a system ID (UUIDv4). This table is also highly restricted and is used exclusively for analytics and reconciliation, not for operational transactions.

### 1.3 UUIDv4 Mapping Table for Analytics

To enable CON-23A501C051 (correlate donor impact receipts with beneficiary redemption events without linking PII), a mapping table is established.

Table Name: pii_analytics_mapping
Fields:
beneficiary_id (UUIDv4): The unique system identifier for the beneficiary.
hash_value (VARCHAR(64)): The SHA-256 hash of the beneficiary's PII (e.g., legal name + demographic status).
salt_version (UUIDv4): The version of the salt used to generate the hash.
created_at (TIMESTAMPTZ): Timestamp of mapping creation.
is_active (BOOLEAN): Flag to indicate if the mapping is currently valid (for salt rotation purposes).

### 1.4 API Contract Implications

- Registration: The `POST /api/v1/beneficiaries/register` endpoint must accept raw PII, hash it using the current salt, store the raw PII in the pii_raw schema, and return the hash_value and beneficiary_id (UUIDv4) in the response.
- Redemption: The `POST /api/v1/redemptions/scan` endpoint must accept the beneficiary_hash (not raw PII) to validate eligibility and process the transaction. The financial ledger will record the hash_value, not the PII.

We will implement a Multi-AZ Aurora PostgreSQL Cluster per Metropolitan Region strategy. This ensures that data generated by users in a specific region never leaves that region's physical infrastructure boundary unless explicitly routed for cross-region replication (which is restricted for PII).

### 2.2 Cross-Border Data Residency Compliance

For future expansion beyond the initial US metro footprints ([CON-9B82D67FAF](../project_glossary.md#con-9b82d67faf), [CON-DDB51EBF45](../project_glossary.md#con-ddb51ebf45)), the architecture must support Cross-Border Data Residency Compliance. This requires:

Geofencing at the API Layer: The API Orchestration Layer ([SUR-85E4A5B6E7](../project_glossary.md#sur-85e4a5b6e7)) must enforce strict geofencing rules. User registration requests must be validated against the user's IP address and/or GPS coordinates to determine the correct jurisdictional cluster.
Data Sovereignty Enforcement: Cross-region replication of PII (Beneficiary legal names, demographic status) is strictly prohibited unless explicit legal consent is obtained and the target region has a data sharing agreement with the source region. This is enforced by the Identity & Access Management (CAP-IDENTITY-ACCESS-MANAGEMENT) service.
Local Data Processing: All PII processing (hashing, anonymization) must occur within the jurisdictional cluster. No PII should be transmitted to a central analytics cluster for processing.

### 2.4 Data Isolation and Access Control

To enforce strict data isolation (CON-0A0288EED4, CON-92F07E31B0), the following access control measures will be implemented:

Schema-Level Isolation: Each jurisdictional cluster will have separate schemas for pii_data, financial_ledger, and analytics_data. The pii_data schema will be accessible only by the Identity & Access Management service.
Row-Level Security (RLS): RLS policies will be enforced to ensure that application services can only access data relevant to their jurisdiction. For example, the NYC-based Merchant POS integration service will only be able to query the NYC cluster.
Network Isolation: Each Aurora cluster will reside in a private subnet within its respective VPC. Public access is disabled. Access is granted only via VPC endpoints from the application services running in the same region.

### 2.5 Disaster Recovery and Business Continuity

Recovery Time Objective (RTO): ASSUMPTION: 15 minutes - reversible pending governance owner confirmation. This target is not established in the project requirement and requires explicit governance owner confirmation.
Recovery Point Objective (RPO): ASSUMPTION: 5 minutes - reversible pending governance owner confirmation. This target is not established in the project requirement and requires explicit governance owner confirmation.
Cross-Region Disaster Recovery: In the event of a regional outage, a secondary standby cluster will be maintained in a different AWS region (e.g., us-west-1 for SF, us-east-2 for NYC, us-west-2 for CHI). This standby cluster will be updated via asynchronous replication from the primary cluster. Note: PII data replication to the standby cluster is subject to cross-border data residency constraints and may require manual intervention or legal review for activation.

## 3. FTC Anonymity Controls and De-anonymization Prevention

This section defines the technical controls to adhere to FTC guidelines on anonymity, ensuring no de-anonymization attacks can link beneficiaries to donors through metadata analysis. It establishes strict data retention policies for donor transaction history versus anonymous redemption analytics.

### 3.1 Metadata Sanitization and PII Stripping

To prevent de-anonymization, all data flowing from the Donor Onboarding & Funding Activation journey ([JNY-62D850E94B](../project_glossary.md#jny-62d850e94b)) to the anonymous redemption analytics pipeline must undergo rigorous metadata stripping. The system must ensure that no direct or indirect identifiers (names, email addresses, IP addresses, device fingerprints) are stored in the analytics data lake or the financial ledger (SUR-FA61592CD4).

Control 1: IP Address Masking: All incoming requests to the API Orchestration Layer (SUR-85E4A5B6E7) must have their source IP addresses masked or truncated to the /24 subnet before being logged or stored. This prevents geolocation-based de-anonymization of donors or beneficiaries.
Control 2: Device Fingerprint Exclusion: The Expo mobile application (Expo v51 / React Native) must not transmit device-specific identifiers (e.g., IMEI, Android ID, IDFV) to the backend. Only anonymous, rotation-based session tokens stored in SecureStore ([CON-34312C6DC9](../project_glossary.md#con-34312c6dc9)) are permitted for session management.
Control 3: Timestamp Normalization: Transaction timestamps must be rounded to the nearest minute in the analytics layer to prevent precise correlation attacks between donor funding events and beneficiary redemption events.

### 3.2 Donor-Beneficiary Linkage Prevention

The core mission of MealCredit is to decouple food assistance from social stigma. Therefore, the system must architecturally prevent any linkage between a specific donor and a specific beneficiary.

Control 4: Decoupled Event Processing: Donor funding events and beneficiary redemption events are processed as independent, asynchronous streams. The Financial Ledger (SUR-FA61592CD4) does not store a foreign key linking a donor ID to a beneficiary ID. Instead, it tracks aggregate credit pool utilization ([CON-2059B17FB2](../project_glossary.md#con-2059b17fb2)) and Donation-to-Redemption Velocity (DRV) ([CON-D0F5814F21](../project_glossary.md#con-d0f5814f21)) at the metro footprint level (SF, NYC, Chicago).
Control 5: UUIDv4 Mapping for Analytics: As defined in the PII Segregation strategy, beneficiary identity is hashed using SHA-256 with a rotating salt. The pii_analytics_mapping table is the only repository of this link, and it is strictly isolated from the financial ledger. Analytics queries must only join on the hash value, never on raw PII.
Control 6: Aggregated Impact Reporting: Donor impact receipts (CON-23A501C051) must be generated using aggregated data. For example, a donor receipt might state "Your donation provided 50 meals in San Francisco," but it must never specify which beneficiaries received those meals or their demographic details.

### 3.3 Strict Data Retention Policies

To minimize the risk of de-anonymization over time, strict data retention policies must be enforced, differentiating between donor transaction history and anonymous redemption analytics.

Control 7: Donor Transaction History Retention: Donor transaction history (payment details, donation amounts, dates) must be retained for a period defined by financial regulations and tax reporting requirements. However, this data must be stored in a separate, highly restricted schema accessible only by the Platform Administrator ([ACT-086A974D63](../project_glossary.md#act-086a974d63)) and the Compliance, Security & Audit capability ([CAP-COMPLIANCE-SECURITY-AUDIT](../project_glossary.md#cap-compliance-security-audit)).
Control 8: Beneficiary PII Retention: Beneficiary PII (legal names, demographic status) must be retained only for the duration of their active eligibility period as defined by the NGO Operator ([ACT-09E028AEB0](../project_glossary.md#act-09e028aeb0)). Upon offboarding ([JNY-4C4BA15817](../project_glossary.md#jny-4c4ba15817)), all raw PII must be cryptographically erased (not just soft-deleted) from the beneficiaries table. The hashed identifier must be retained for analytics purposes.
Control 9: Redemption Analytics Retention: Anonymous redemption analytics (hash values, transaction amounts, timestamps rounded to the minute) must be retained for a period defined by trend analysis and liquidity planning requirements. After this period, this data must be aggregated into yearly summaries and the raw records purged.
Control 10: Log Retention: Application logs containing any potential PII (e.g., error messages, stack traces) must be retained for a period defined by operational audit requirements and then automatically purged. Logs must be scanned for PII patterns before storage, and any matches must be redacted.

### 3.4 De-anonymization Attack Mitigation

The system must be designed to resist common de-anonymization attacks, such as linkage attacks, inference attacks, and differential privacy attacks.

Control 11: K-Anonymity for Analytics: When generating reports for NGO Operators or Platform Administrators, the analytics data must satisfy k-anonymity with a threshold defined by statistical significance requirements for analytics. This means that any report must include data for a minimum number of distinct beneficiaries to prevent individual identification.
Control 12: Differential Privacy for Public Reports: Any public-facing reports or dashboards must apply differential privacy noise to the data. This ensures that the inclusion or exclusion of any single donor's contribution or beneficiary's redemption does not significantly alter the statistical output.
Control 13: Regular De-anonymization Audits: The Compliance, Security & Audit capability (CAP-COMPLIANCE-SECURITY-AUDIT) must conduct regular audits to test the system's resistance to de-anonymization attacks. These audits should include simulated linkage attacks using external public datasets.

### 3.8 Token Structure and Cryptographic Signing

To support the Beneficiary Eligibility & Voucher Redemption journey ([JNY-E82B8A88D8](../project_glossary.md#jny-e82b8a88d8)) during network outages, the system issues offline tokens. These tokens must be self-contained, verifiable, and tamper-proof.

Token Payload Schema:
The offline token is a JSON object, signed using an asymmetric key pair (RSASSA-PKCS1-v1_5 with SHA-256). The payload includes:

| Field | Type | Description |
| :--- | :--- | :--- |
| `token_id` | UUIDv4 | Unique identifier for this specific token instance. |
| `beneficiary_id` | UUIDv4 | Hashed identifier of the beneficiary (see PII Segregation Strategy). |
| `credit_amount` | Decimal | The monetary value of the credits available in this token. |
| `expiry_timestamp` | Unix Epoch | The exact time (UTC) after which the token is invalid. |
| `nonce` | String (Base64) | A cryptographically random value to prevent identical payloads from producing identical signatures, mitigating replay attacks across different redemption attempts. |
| `signature` | String (Base64) | The RSASSA-PKCS1-v1_5 signature of the canonical JSON payload (excluding the signature field itself). |

Signing Process:
1. The `Token Issuance Service` (owned by Financial Ledger Data Model) constructs the payload.
2. The payload is canonicalized (sorted keys, no whitespace).
3. The private key (stored in AWS KMS) signs the canonical payload.
4. The signature is appended to the payload, and the entire object is encoded as a QR code.

Verification Process (Merchant POS):
1. The Merchant POS app (Expo) scans the QR code and decodes the JSON.
2. The app retrieves the public key from a cached, periodically updated bundle (synced when online).
3. The app verifies the signature using the public key.
4. The app checks `expiry_timestamp` against its local clock.
5. The app checks `nonce` against a local cache of recently used nonces to prevent immediate replay.

### 3.9 Time-Bound Expiration and Replay Protection

To satisfy [CON-3335D67672](../project_glossary.md#con-3335d67672), tokens must have a short validity window to limit the attack surface for replay attacks.

Expiration Policy:
- Token Lifetime: Tokens are valid for a maximum of 15 minutes from issuance. This balances the need for offline usability with security.
- Clock Skew Tolerance: The Merchant POS app must allow a clock skew tolerance of ±30 seconds. If the local clock is significantly skewed, the app should prompt the user to sync time via NTP or disable offline mode.

Replay Attack Mitigation:
- Nonce Uniqueness: Each token must have a unique nonce. The Merchant POS app maintains a local LRU (Least Recently Used) cache of the last 1,000 nonces. If a scanned token's nonce is in the cache, the redemption is rejected.
- Server-Side Validation: Upon reconnection, the Merchant POS app syncs its local redemption log with the `Financial Ledger Data Model`. The server validates that each nonce was only used once. If a duplicate nonce is detected, the server flags the merchant account for review (see Merchant Payout Failure & Error Handling).

### 3.10 Secure Client-Side Storage (Expo SecureStore)

To satisfy CON-34312C6DC9, offline tokens and cryptographic keys must be stored securely on the device, preventing theft or cloning.

SecureStore Implementation:
- Key Storage: The public key bundle and the merchant's private signing key (if applicable for two-way authentication) are stored in Expo's SecureStore. This leverages the device's hardware-backed keystore (Android Keystore / iOS Keychain).
- Token Storage: Offline tokens are stored in SecureStore with a short TTL. They are automatically purged from memory when the app is backgrounded or closed.
- Access Control: Access to SecureStore is restricted to the app's sandbox. No other app on the device can access these keys.

Threat Model:
- Device Compromise: If the device is rooted/jailbroken, SecureStore may be vulnerable. Mitigation: Implement certificate pinning and detect root/jailbreak status. If detected, disable offline mode and require online authentication.
- Token Cloning: Since tokens are signed and nonces are unique, cloning a token does not allow an attacker to redeem it twice. The first redemption consumes the nonce.

### 3.11 Offline Fallback Interface Design

To satisfy [CON-387CDD9AEB](../project_glossary.md#con-387cdd9aeb), the offline fallback interface must be intuitive and accessible.

UI/UX Requirements:
- Visual Clarity: The QR code must be large and high-contrast, ensuring it can be scanned by standard POS scanners even in low-light conditions.
- Status Indicators: The app must clearly display the offline status (e.g., "Offline Mode Active") and the token's expiration time.
- Accessibility: The interface must support screen readers (VoiceOver/TalkBack) and high-contrast modes, as per [CON-68497304B1](../project_glossary.md#con-68497304b1).

### 3.12 Key Management and Rotation

Key Rotation Strategy:
- Public Key Updates: The public key bundle is updated daily via a background sync when the device is online. The app caches the last 3 bundles to allow verification of tokens issued during the transition period.
- Private Key Rotation: The private key is rotated quarterly. During rotation, a grace period of 24 hours is allowed where both old and new keys are accepted for verification.

Knowledge Gaps:
- `KNOWLEDGE_GAP: offline_token_security - The exact algorithm for nonce generation (e.g., UUIDv4 vs. random bytes) must be established by the Security Team to ensure cryptographic strength.`
- `KNOWLEDGE_GAP: offline_token_security - The maximum number of concurrent offline tokens per beneficiary must be defined by the Product Team to prevent credit hoarding.`

## 4. Append-Only Cryptographic Log Auditing Architecture

This section defines the architectural contract for the append-only cryptographic log auditing system within the Aurora PostgreSQL Data Persistence Layer (SUR-FA61592CD4). This system is the primary control mechanism for SOC2 Type II structural planning ([CON-81FB01F06B](../project_glossary.md#con-81fb01f06b), [CON-E84412A0FA](../project_glossary.md#con-e84412a0fa)) and ensures the integrity of all financial ledger mutations ([CON-1762EA5021](../project_glossary.md#con-1762ea5021), [CON-6061FCCA83](../project_glossary.md#con-6061fcca83)). It operates in strict conjunction with the PII Segregation & Anonymization Strategy to ensure that audit logs never contain raw beneficiary PII, relying instead on the cryptographic hashes defined in the pii_analytics_mapping table.

### 4.1 Audit Log Schema & Data Model

The audit log is implemented as a dedicated, append-only table group within Aurora PostgreSQL. To enforce immutability and prevent internal tampering, this table group is isolated via strict database-level permissions and AWS IAM integration.

Table: `audit.financial_ledger_mutation_log`

| Column | Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `log_id` | UUID | `PRIMARY KEY`, `DEFAULT gen_random_uuid()` | Unique identifier for the audit log entry. |
| `mutation_id` | UUID | `NOT NULL` | Reference to the specific financial transaction or ledger mutation in the financial_ledger table. |
| `actor_id` | UUID | `NOT NULL` | Reference to the actor (Donor, Beneficiary, Merchant, NGO Operator, Platform Administrator) initiating the mutation. |
| `action_type` | VARCHAR(50) | `NOT NULL` | Enumerated action: CREDIT_ISSUANCE, CREDIT_REDEMPTION, CREDIT_REFUND, NGO_ALLOCATION, PLATFORM_ADJUSTMENT. |
| `payload_hash` | CHAR(64) | `NOT NULL` | SHA-256 hash of the pre-mutation and post-mutation state payload. Used for integrity verification. |
| `previous_state_hash` | CHAR(64) | `NOT NULL` | SHA-256 hash of the state immediately preceding the mutation. |
| `current_state_hash` | CHAR(64) | `NOT NULL` | SHA-256 hash of the state immediately following the mutation. |
| `chain_hash` | CHAR(64) | `NOT NULL` | SHA-256 hash of the `log_id` + `mutation_id` + `payload_hash` + `previous_chain_hash`. Creates the cryptographic chain. |
| `previous_chain_hash` | CHAR(64) | `NOT NULL` | The `chain_hash` of the immediately preceding log entry. For the first entry, this is a fixed genesis hash. |
| `timestamp_utc` | TIMESTAMPTZ | `DEFAULT now()`, `NOT NULL` | UTC timestamp of the mutation. |
| `metadata_json` | JSONB | `DEFAULT '{}'` | Non-PII metadata: `geo_region` (e.g., us_sfbay), `pos_terminal_id`, `device_fingerprint_hash`. |
| `cloudtrail_event_id` | VARCHAR(128) | `NULL` | AWS CloudTrail Event ID associated with the infrastructure change or API call that triggered the mutation. |

Integrity Constraint:
A CHECK constraint must be enforced at the database level to ensure `chain_hash = SHA256(log_id || mutation_id || payload_hash || previous_chain_hash)`. This ensures that any modification to a prior log entry invalidates the chain, making tampering immediately detectable.

### 4.3 AWS CloudTrail Integration

To ensure infrastructure-level auditability, all mutations to the `audit.financial_ledger_mutation_log` table are logged via AWS CloudTrail. This provides an external, immutable record of who accessed or modified the audit logs themselves.

1. CloudTrail Event Capture: AWS CloudTrail is configured to log all PutObject, UpdateTable, and Insert operations on the Aurora PostgreSQL cluster. These events are stored in a dedicated S3 bucket with versioning and encryption enabled.
2. Event Correlation: The `cloudtrail_event_id` field in the `audit.financial_ledger_mutation_log` table is populated with the corresponding CloudTrail Event ID for the API call that triggered the mutation. This allows auditors to trace a financial mutation back to the specific API call and IAM role that initiated it.
3. Access Control: Only the `Platform Administrator` (ACT-086A974D63) and the `SOC2 Compliance Auditor` (a temporary role created for audit periods) have read access to the CloudTrail logs. All other actors, including `NGO Operators` (ACT-09E028AEB0) and `Platform Administrators` during normal operations, are denied access to the raw CloudTrail data.

### 4.4 SOC2 Type II Structural Planning

This architecture is designed to meet the following SOC2 Type II Trust Service Criteria:

- CC6.1 (Logical and Physical Access Controls): The `audit.financial_ledger_mutation_log` table is restricted to INSERT operations only for application service accounts. SELECT operations are restricted to the `SOC2 Compliance Auditor` role. AWS IAM policies enforce these restrictions at the database level.
- CC6.6 (System Boundary): The append-only log is part of the system boundary. Its immutability is enforced by both database constraints and AWS CloudTrail logging.
- CC7.2 (System Monitoring): The hourly Lambda validation function serves as a continuous monitoring control. It detects any tampering with the audit log chain in near real-time.
- CC8.1 (Change Management): All changes to the audit log schema or hashing algorithm require a formal change request approved by the `Platform Administrator` (ACT-086A974D63) and documented in the SOC2 change management log.

### 4.5 PII Segregation in Audit Logs

To ensure compliance with the PII Segregation & Anonymization Strategy, the audit log must never contain raw beneficiary PII.

1. No PII Fields: The `audit.financial_ledger_mutation_log` table contains no fields for beneficiary legal names, demographic status, or contact information.
2. Hashed References: All references to beneficiaries are via the `actor_id` field, which is a UUID. The mapping between `actor_id` and the beneficiary's PII is stored exclusively in the `pii_analytics_mapping` table, which is cryptographically segregated from the audit log.
3. Payload Hashing: The `payload_hash` field contains a hash of the pre- and post-mutation state. This hash is computed after PII has been stripped or hashed, ensuring that the audit log itself never contains raw PII, even in hashed form, unless it is part of the `pii_analytics_mapping` table.

### 4.6 Implementation Notes for Development

1. Database Migration: Create the `audit` schema and `financial_ledger_mutation_log` table as part of the initial Aurora PostgreSQL migration.
2. Application Layer: The `Financial Ledger` service must compute the `payload_hash`, `previous_state_hash`, and `current_state_hash` for every mutation and insert the record into the `audit.financial_ledger_mutation_log` table within the same transaction as the ledger mutation.
3. Lambda Function: Deploy the `audit_chain_validator` Lambda function with a CloudWatch Events rule to trigger it hourly. The function must have read-only access to the `audit` schema.
4. CloudTrail: Enable CloudTrail for the Aurora PostgreSQL cluster and configure the S3 bucket for long-term storage. Ensure the `cloudtrail_event_id` is captured in the application logs and correlated with the audit log entry.
5. IAM Policies: Create IAM policies for the `SOC2 Compliance Auditor` role that grant read-only access to the `audit` schema and CloudTrail logs. Deny all other actors access to these resources.
