# Distributed Tracing & Log Aggregation

## 1. OpenTelemetry SDK Configuration and Instrumentation Strategy

This section defines the OpenTelemetry (OTel) SDK configuration and instrumentation strategy for the MealCredit serverless architecture. It ensures end-to-end trace visibility for the Platform Administrator and NGO Operator services, maintaining strict data isolation and low-latency requirements.

### 1.1 Core SDK Configuration and Context Propagation

The OTel SDK will be configured to use the W3C Trace Context standard for cross-service trace propagation. This ensures that trace context is preserved across HTTP requests (API Gateway to Lambda) and database calls (Aurora PostgreSQL).

- **Propagators:** The SDK must be initialized with TraceContextPropagator and BaggagePropagator to handle standard HTTP headers (traceparent, tracestate) and custom baggage for tenant isolation.
- **Resource Detection:** A custom ResourceDetector will be implemented to inject project-specific attributes, including service.name (e.g., mealcredit-platform-admin, mealcredit-ngo-operator), deployment.environment (e.g., prod-sf, prod-nyc), and tenant.id (for multi-tenant isolation).
- **Span Processor:** To minimize cold-start latency and avoid blocking the main execution thread, the BatchSpanProcessor will be used with a schedule_delay of 1 second and a max_export_batch_size of 64. This ensures traces are exported asynchronously without impacting the p99 latency target of 250ms.

### 1.2 HTTP Request Instrumentation

All HTTP requests originating from the Platform Administrator and NGO Operator services must inject trace context to enable distributed tracing across the API Orchestration Layer and downstream services.

- **HTTP Client Instrumentation:** The OTel HTTP client middleware will be configured to automatically inject traceparent and tracestate headers into all outgoing HTTP requests. This includes requests to the Stripe Issuing Proxy, gRPC services, and internal API Gateway endpoints.
- **Context Extraction:** Incoming HTTP requests (e.g., from API Gateway or Stripe Webhooks) will have their trace context extracted using the TraceContextPropagator. If a traceparent header is present, it will be used to continue the existing trace. If not, a new root span will be created.
- **Error Handling:** HTTP errors (4xx, 5xx) will be recorded as span events with the status code and error message, ensuring that failures are visible in the trace.

### 1.5 Integration with Observability Backend

The OTel SDK will be configured to export traces to the project's observability backend (e.g., AWS X-Ray or OpenSearch).

- **Exporter Configuration:** The OTLPExporter will be configured to send traces to the observability backend's OTLP endpoint. The exporter will be configured with a timeout of 10 seconds and a retry policy to handle transient network failures.
- **Log Correlation:** Logs generated by the Platform Administrator and NGO Operator services will be correlated with traces using the trace_id and span_id from the current OTel context. This ensures that logs can be easily viewed in the context of the trace.

## 2. AWS X-Ray Segments and CloudWatch Logs Data Schemas

This section defines the concrete data schemas, field mappings, and structural requirements for AWS X-Ray segments and AWS CloudWatch Logs. These schemas are engineered to capture the high-fidelity, immutable audit trail required for SOC2 Type II compliance ([CON-81FB01F06B](../project_glossary.md#con-81fb01f06b), [CON-E84412A0FA](../project_glossary.md#con-e84412a0fa)) and to support the append-only cryptographic log auditing mandated for the Aurora PostgreSQL financial ledger ([CON-6061FCCA83](../project_glossary.md#con-6061fcca83), [CON-1762EA5021](../project_glossary.md#con-1762ea5021)). The schemas ensure that all financial ledger mutations and payment processing events from the Transaction & Financial Engine ([CAP-TRANSACTION-FINANCIAL-ENGINE](../project_glossary.md#cap-transaction-financial-engine)) are captured with sufficient granularity for forensic analysis and regulatory reporting.

### 2.1 AWS X-Ray Segment Schema: Financial Transaction Trace

The X-Ray segment schema is designed to capture the end-to-end lifecycle of a financial transaction, from the initial Donor ([ACT-80C62C7814](../project_glossary.md#act-80c62c7814)) donation round-up to the final Beneficiary ([ACT-ADA6716160](../project_glossary.md#act-ada6716160)) redemption at a Merchant ([ACT-AF904DCFF9](../project_glossary.md#act-af904dcff9)). This schema prioritizes traceability and latency monitoring ([CON-06232374D9](../project_glossary.md#con-06232374d9), [CON-4152F2C7C3](../project_glossary.md#con-4152f2c7c3)).

**Segment Root Properties:**
- `id`: UUIDv4, unique identifier for the trace.
- `name`: String, service name (e.g., mealcredit-transaction-engine).
- `start_time`: Float, epoch timestamp in seconds.
- `end_time`: Float, epoch timestamp in seconds.
- `trace_id`: String, W3C Trace Context compliant trace ID.
- `parent_id`: String, UUIDv4 of the parent segment (if applicable).
- `http`: Object, containing request and response objects for HTTP-based interactions (e.g., Stripe Webhooks, API Gateway calls).
- `subsegments`: Array of subsegments representing downstream service calls (e.g., Aurora PostgreSQL, Redis, Stripe API).

**Mandatory Financial Metadata (Annotations):**
Annotations are indexed and queryable in X-Ray. The following fields are mandatory for all financial transaction traces:
- `transaction_id`: UUIDv4, unique identifier for the financial transaction.
- `transaction_type`: String, enum: DONATION, CREDIT_ALLOCATION, REDEMPTION, REFUND, PAYOUT.
- `donor_id_hash`: String, SHA-256 hash of the Donor ID (ensures anonymity per [CON-B3D71A437D](../project_glossary.md#con-b3d71a437d)).
- `beneficiary_id_hash`: String, SHA-256 hash of the Beneficiary ID (ensures anonymity per CON-B3D71A437D).
- `merchant_id`: String, unique identifier for the Merchant.
- `amount_cents`: Integer, transaction amount in cents.
- `currency`: String, ISO 4217 currency code (e.g., USD).
- `metro_region`: String, enum: SF, NYC, CHI.
- `compliance_status`: String, enum: PASSED, FAILED, PENDING_REVIEW.

**Mandatory Financial Metadata (Attributes):**
Attributes are not indexed but are stored with the segment. These are used for detailed forensic analysis.
- `ledger_entry_id`: String, unique identifier for the corresponding entry in the append-only ledger.
- `stripe_charge_id`: String, Stripe API charge ID for the transaction.
- `offline_token_id`: String, ID of the offline QR/barcode token if the redemption was offline.
- `replay_attack_check`: Boolean, result of the replay attack check for offline tokens ([CON-3335D67672](../project_glossary.md#con-3335d67672)).
- `pci_dss_compliance_flag`: Boolean, must be true for all traces ([CON-66390130AA](../project_glossary.md#con-66390130aa)).

**Subsegment Schema for Aurora PostgreSQL:**
- `id`: UUIDv4.
- `name`: String, aurora-postgresql.
- `start_time`: Float.
- `end_time`: Float.
- `subsegments`: Array of subsegments for individual SQL statements.
- `sql_statements`: Array of objects, each containing:
  - `statement`: String, the SQL statement executed (parameterized, no PII).
  - `database_type`: String, PostgreSQL.
  - `database_version`: String, version of Aurora PostgreSQL.
  - `table_name`: String, name of the table accessed (e.g., financial_ledger, credit_pools).
  - `execution_time_ms`: Float, time taken to execute the statement.
  - `rows_affected`: Integer, number of rows affected.
  - `is_append_only`: Boolean, true if the operation was an append-only ledger mutation.

**Subsegment Schema for Stripe API:**
- `id`: UUIDv4.
- `name`: String, stripe-api.
- `start_time`: Float.
- `end_time`: Float.
- `http`: Object, containing request and response objects.
- `stripe_api_version`: String, version of the Stripe API called.
- `stripe_error_code`: String, Stripe error code if the request failed.

### 2.2 AWS CloudWatch Logs Schema: Financial Ledger Audit

CloudWatch Logs will serve as the primary storage for immutable, append-only financial ledger mutations. The log schema is designed to be machine-readable, structured, and compliant with SOC2 Type II requirements for data integrity and non-repudiation.

**Log Group Structure:**
- `/mealcredit/financial-ledger/audit`: Primary log group for all financial ledger mutations.
- `/mealcredit/payment-processing/events`: Log group for Stripe webhook events and payment processing callbacks.
- `/mealcredit/compliance/fraud-investigation`: Log group for flagged transactions and fraud investigation events.

**Log Stream Naming Convention:**
- `{service-name}/{date}/{hour}`: e.g., `transaction-engine/2024-05-20/14`.

**Log Event Schema (JSON):**
Each log event is a JSON object with the following mandatory fields:
- `timestamp`: ISO 8601 timestamp of the event.
- `event_id`: UUIDv4, unique identifier for the log event.
- `event_type`: String, enum: LEDGER_MUTATION, PAYMENT_CALLBACK, FLAGGED_FOR_REVIEW, AUDIT_LOG_EXPORT.
- `transaction_id`: UUIDv4, unique identifier for the financial transaction.
- `mutation_type`: String, enum: CREATE, UPDATE, DELETE (note: DELETE is strictly prohibited for financial ledger entries, only CREATE and UPDATE are allowed).
- `ledger_entry_id`: String, unique identifier for the ledger entry.
- `previous_state_hash`: String, SHA-256 hash of the previous state of the ledger entry (for integrity verification).
- `new_state_hash`: String, SHA-256 hash of the new state of the ledger entry.
- `actor_id_hash`: String, SHA-256 hash of the actor ID (Donor, Beneficiary, Merchant, NGO Operator) associated with the transaction.
- `amount_cents`: Integer, transaction amount in cents.
- `currency`: String, ISO 4217 currency code.
- `metro_region`: String, enum: SF, NYC, CHI.
- `metadata`: Object, additional context-specific metadata.

### 2.3 Data Isolation and PII Sanitization

To ensure strict compliance with FTC guidelines on anonymity (CON-B3D71A437D) and data isolation ([CON-0A0288EED4](../project_glossary.md#con-0a0288eed4)), all PII must be sanitized before being written to X-Ray or CloudWatch Logs.

**Sanitization Rules:**
- **Beneficiary Legal Names:** Must be replaced with beneficiary_id_hash (SHA-256).
- **Donor PII:** Must be replaced with donor_id_hash (SHA-256).
- **Beneficiary Demographic Status:** Must be replaced with beneficiary_id_hash (SHA-256).
- **Credit Pool Utilization Rate:** Must be logged as a metric, not as raw PII ([CON-2059B17FB2](../project_glossary.md#con-2059b17fb2)).
- **PCI-DSS Compliance:** No raw card data (PAN, CVV, etc.) is ever logged. All card data interactions are handled by Stripe Elements, and only Stripe API references (e.g., stripe_charge_id) are logged (CON-66390130AA).

**Implementation Notes:**
- A dedicated PIISanitizer middleware will be implemented in the API Orchestration Layer ([SUR-85E4A5B6E7](../project_glossary.md#sur-85e4a5b6e7)) to intercept all log events and apply the sanitization rules before they are forwarded to X-Ray or CloudWatch.
- The PIISanitizer will use a configurable mapping of PII fields to their corresponding hash fields to ensure consistency across all services.

### 3.1 Trace Context Propagation Strategy
To maintain end-to-end visibility across the asynchronous serverless architecture, the platform will enforce W3C Trace Context propagation for all inter-service communication. This ensures that a single transaction, such as a Beneficiary redemption or a Donor payout, can be traced from the initial API Gateway request through to downstream Lambda functions, SQS queues, and EventBridge events.

- **HTTP/gRPC Interfaces:** All HTTP requests (API Gateway to Lambda) and gRPC calls (API Orchestration to Financial Engine) MUST inject the traceparent and tracestate headers. The traceparent header will follow the W3C format: version-trace_id-span_id-trace_flags.
- **SQS Queues:** SQS message attributes MUST include traceparent and tracestate as string attributes. When a Lambda function processes a message from an SQS queue, it MUST extract the traceparent attribute to resume the existing trace context rather than starting a new root span.
- **EventBridge Events:** EventBridge events MUST include the traceparent attribute in the event detail or metadata. The consuming Lambda function will extract this to link the event to the originating trace.

### 3.2 Asynchronous Event Correlation Patterns
The platform relies heavily on asynchronous events for financial reconciliation and payout processing. To ensure these events are correlated correctly:

- **Correlation ID Injection:** Every incoming request to the API Orchestration Layer MUST generate a unique correlation_id (UUIDv4). This ID MUST be propagated to all downstream SQS messages and EventBridge events.
- **SQS Message Structure:** SQS messages MUST include the following standard attributes:
  - `traceparent`: W3C Trace Context string.
  - `correlation_id`: UUIDv4 string for application-level correlation.
  - `event_type`: String identifying the business event (e.g., PAYOUT_INITIATED, RECONCILIATION_BATCH_COMPLETE).
- **EventBridge Event Structure:** EventBridge events MUST include:
  - `traceparent`: W3C Trace Context string.
  - `correlation_id`: UUIDv4 string.
  - `source`: Identifies the originating service (e.g., mealcredit.financial-engine).
  - `detail-type`: Identifies the specific event type (e.g., MerchantPayoutError).

### 3.3 Lambda Function Trace Context Handling
Lambda functions MUST be configured to automatically extract and inject trace context where possible, and manually handle it for asynchronous events.

- **Synchronous Invocations:** For Lambda functions invoked synchronously via API Gateway or gRPC, the runtime MUST automatically extract the traceparent header and create a child span under the parent context.
- **Asynchronous Invocations (SQS/EventBridge):** For Lambda functions triggered by SQS or EventBridge, the function code MUST explicitly extract the traceparent attribute from the event payload and use it to initialize the trace context. This ensures that the span created by the Lambda is a child of the originating trace.
- **Error Handling:** If a Lambda function encounters an error, it MUST log the traceparent and correlation_id in the error log. This allows for immediate correlation of the error with the originating trace in the log aggregation system.

## 4. Architectural Overview

This document defines the distributed tracing and log aggregation strategy for the MealCredit platform. The design ensures end-to-end visibility across the serverless architecture (AWS Lambda, API Gateway, Aurora PostgreSQL, Redis, Stripe) while strictly adhering to PCI-DSS Level 1, SOC2 Type II, and FTC Anonymity Guidelines. The architecture prioritizes low-latency trace propagation for real-time POS clearance and immutable audit trails for financial reconciliation.

### 4.1 Core Design Principles

1.  **Zero-PII in Traces:** Trace context (traceparent, spanid) and operational metadata are propagated freely. PII (Beneficiary names, donor PII, raw card data) is NEVER included in trace attributes or log events. PII is masked or hashed at the ingestion layer before persistence.
2.  **W3C Trace Context Standard:** All services adhere to the W3C Trace Context standard (`traceparent` header) to ensure interoperability between AWS services, third-party webhooks (Stripe), and the Expo mobile client.
3.  **Immutable Financial Audit:** Financial ledger mutations are logged to an append-only cryptographic log in Aurora PostgreSQL, with trace IDs correlated to the operational logs in CloudWatch/OpenSearch for forensic reconciliation.
4.  **Role-Based Visibility:** Dashboard access is strictly governed by the Access Control & Multi-Tenant Isolation contract. NGO Operators see only their organization's data; Platform Administrators have global visibility.

### 5.1 Propagation Across Service Boundaries

Trace context must be preserved and injected at every service boundary to enable end-to-end correlation.

1.  **Expo Mobile Client to API Gateway:**
    *   The Expo mobile application (using React Native/Expo) injects the `traceparent` header into all HTTP requests via the API Gateway.
    *   **Implementation:** The mobile SDK initializes an OpenTelemetry tracer. On every API call, the current span context is serialized into the W3C Trace Context format and attached to the request headers.
    *   **Constraint:** No PII (e.g., beneficiary email, donor ID) is included in the trace headers. Only the `traceparent` and a non-identifying `correlation_id` (UUIDv4) are propagated.

2.  **API Gateway to Lambda Functions:**
    *   API Gateway is configured to extract the `traceparent` header from incoming requests and pass it to the invoked Lambda function.
    *   **Implementation:** Lambda functions use the AWS X-Ray SDK (or OpenTelemetry AWS SDK) to create a new child span, inheriting the parent trace ID. If no `traceparent` is present (e.g., internal scheduled events), a new trace is initiated.

3.  **Lambda Functions to Aurora PostgreSQL:**
    *   Database queries are instrumented to capture execution time, query plan, and errors within the active trace.
    *   **Implementation:** The Aurora PostgreSQL driver is configured with the AWS X-Ray SDK for RDS. The `traceparent` is passed as a connection parameter or query hint, allowing the database layer to associate query latency with the specific user request.

4.  **Lambda Functions to Stripe Webhooks:**
    *   Trace context is extracted from incoming Stripe webhook payloads (if present) to correlate payment events with user actions.
    *   **Implementation:** When the platform initiates a Stripe API call (e.g., creating a virtual card), the `traceparent` is included in the Stripe request headers. Stripe's response is logged with the same trace context, enabling correlation of payment success/failure with the originating user action.

5.  **Asynchronous Event Propagation (EventBridge & SQS):**
    *   **EventBridge:** Events published to EventBridge must include the `traceparent` and `correlation_id` in the event detail. EventBridge preserves these attributes, allowing downstream Lambda consumers to resume the trace.
    *   **SQS:** Messages in SQS queues must include the `traceparent` and `correlation_id` in the message attributes. This ensures that asynchronous error handling and retry logic maintain trace continuity.

### 5.2 Merchant Payout Error Handling Flow Correlation

The Merchant Payout Error Handling Flow involves multiple asynchronous steps. To ensure visibility:

1.  **Payout Initiation:** The Financial Reconciliation & Payout Workers service initiates a payout and publishes an event to EventBridge with the `traceparent` and `correlation_id`.
2.  **Payout Processing:** A Lambda function processes the EventBridge event, extracts the `traceparent`, and creates a span. If the payout fails, it publishes an error message to an SQS queue with the same `traceparent` and `correlation_id`.
3.  **Error Handling:** A dedicated error handling Lambda function processes the SQS message, extracts the `traceparent`, and creates a span. It then logs the error with the `traceparent` and `correlation_id`.
4.  **Notification:** The error handling Lambda function may trigger a notification to the Platform Administrator via EventBridge, again propagating the `traceparent` and `correlation_id`.

### 5.3 Financial Reconciliation Process Correlation

The Financial Reconciliation process involves batch processing and periodic checks. To ensure visibility:

1.  **Batch Initiation:** A scheduled Lambda function initiates a reconciliation batch and publishes an event to EventBridge with the `traceparent` and `correlation_id`.
2.  **Batch Processing:** A Lambda function processes the EventBridge event, extracts the `traceparent`, and creates a span. It processes the batch and publishes results to an SQS queue with the same `traceparent` and `correlation_id`.
3.  **Result Processing:** A dedicated result processing Lambda function processes the SQS message, extracts the `traceparent`, and creates a span. It logs the results with the `traceparent` and `correlation_id`.

## 6. Log Aggregation and Ingestion Architecture

### 6.2 Storage Layer

Logs are stored in a tiered architecture to balance cost, performance, and compliance:

1.  **Hot Storage (AWS CloudWatch Logs / OpenSearch):**
    *   **Purpose:** Real-time monitoring, alerting, and short-term debugging.
    *   **Retention:** 90 days for operational logs; 7 days for PII-containing logs.
    *   **Access:** High-speed, low-latency queries for the Platform Administrator ([ACT-086A974D63](../project_glossary.md#act-086a974d63)) and NGO Operator ([ACT-09E028AEB0](../project_glossary.md#act-09e028aeb0)).

2.  **Warm Storage (AWS S3 with Intelligent-Tiering):**
    *   **Purpose:** Long-term audit evidence for SOC2 Type II and financial reconciliation.
    *   **Retention:** 7 years for financial/audit logs; 1 year for operational logs.
    *   **Access:** Batch queries for compliance audits and forensic analysis.
    *   **Archival Trigger:** Logs older than 90 days (operational) or 7 days (PII) are automatically archived to S3.

3.  **Cold Storage (AWS S3 Glacier Deep Archive):**
    *   **Purpose:** Regulatory compliance and disaster recovery.
    *   **Retention:** 7+ years (as required by financial regulations).
    *   **Access:** Retrieval only upon explicit legal or compliance request. Retrieval time: 12-48 hours.

    *   **System Uptime & Availability:** Real-time monitoring of AWS multi-AZ configurations for all critical service endpoints, targeting 99.99% operational uptime ([CON-BF1CD5707E](../project_glossary.md#con-bf1cd5707e), [CON-FD21121DD5](../project_glossary.md#con-fd21121dd5)).

### 7.3 Distributed Tracing Integration

To support these dashboards and alerting rules, distributed tracing will be implemented using OpenTelemetry. Trace context will be propagated across all service boundaries, including:

*   **Expo Mobile Client to API Gateway:** Trace context will be injected into all HTTP requests from the Expo mobile application.
*   **API Gateway to Lambda Functions:** Trace context will be preserved across serverless function invocations.
*   **Lambda Functions to Aurora PostgreSQL:** Database queries will be instrumented to capture execution time and errors within the trace.
*   **Lambda Functions to Stripe Webhooks:** Trace context will be extracted from incoming Stripe webhook payloads to correlate payment events with user actions.

This end-to-end tracing will enable the Platform Administrator to quickly diagnose performance bottlenecks and financial discrepancies, ensuring the platform's reliability and trustworthiness.

### 8.1 Data Classification and Masking Rules

All log data must be classified and processed according to the following sensitivity levels. Masking is applied at the ingestion layer (e.g., Fluent Bit / CloudWatch Agent) before data is persisted to the primary log store.

| Data Classification | Examples | Masking Rule | Retention |
| :--- | :--- | :--- | :--- |
| Highly Sensitive (PII) | Beneficiary legal names, demographic status, donor PII, payment card details (PAN) | Cryptographic Hashing / Irreversible Masking: All PII fields must be hashed using a salted algorithm before logging. Raw PII must never appear in logs. For PCI-DSS compliance, zero raw card data may touch MealCredit servers; logs must only contain Stripe token IDs or last-4 digits if absolutely necessary for debugging (and even then, masked). | 7 Days (Hot Storage) |
| Financial / Audit | Transaction IDs, ledger mutations, payout amounts, donor impact receipts | Immutable Append-Only: Logs must be written to an append-only cryptographic log (CON-1762EA5021) in Aurora PostgreSQL and mirrored to S3 with Object Lock enabled. No field masking is applied to financial IDs to ensure traceability. | 7 Years (Archival) |
| Operational / Debug | API request/response payloads, gRPC service calls, Expo client errors, Next.js server logs | Contextual Masking: Sensitive fields (e.g., user_id, email) are redacted. Trace IDs (W3C Trace Context) are preserved for distributed tracing. | 90 Days (Hot Storage) |
| System / Infrastructure | AWS CloudTrail, VPC Flow Logs, Lambda cold starts, CPU/Memory metrics | No Masking: System logs are generally non-PII. | 365 Days (Hot Storage) |

**KNOWLEDGE_GAP:** The exact hashing algorithm and salt management strategy for PII masking must be established by the Security Architect. **ASSUMPTION:** We will use a consistent, project-wide salt stored in AWS Secrets Manager, rotated annually, to ensure that the same PII always hashes to the same value for cross-log correlation (e.g., linking a Beneficiary's redemption event to their eligibility check) without exposing the raw value.

### 8.2 Log Retention and Archival Strategy

The log aggregation system will utilize a tiered storage strategy to balance cost, performance, and compliance.

1.  **Hot Storage (AWS CloudWatch Logs / OpenSearch):**
    *   **Purpose:** Real-time monitoring, alerting, and short-term debugging.
    *   **Retention:** 90 days for operational logs; 7 days for PII-containing logs.
    *   **Access:** High-speed, low-latency queries for the Platform Administrator (ACT-086A974D63) and NGO Operator (ACT-09E028AEB0).

2.  **Warm Storage (AWS S3 with Intelligent-Tiering):**
    *   **Purpose:** Long-term audit evidence for SOC2 Type II and financial reconciliation.
    *   **Retention:** 7 years for financial/audit logs; 1 year for operational logs.
    *   **Access:** Batch queries for compliance audits and forensic analysis.
    *   **Archival Trigger:** Logs older than 90 days (operational) or 7 days (PII) are automatically archived to S3.

3.  **Cold Storage (AWS S3 Glacier Deep Archive):**
    *   **Purpose:** Regulatory compliance and disaster recovery.
    *   **Retention:** 7+ years (as required by financial regulations).
    *   **Access:** Retrieval only upon explicit legal or compliance request. Retrieval time: 12-48 hours.

**KNOWLEDGE_GAP:** The specific data residency requirements for each metropolitan footprint (SF, NYC, Chicago) must be confirmed. **ASSUMPTION:** All logs generated within a specific metro footprint will be stored in the corresponding AWS Region (e.g., logs from NYC merchants will be stored in us-east-1) to comply with data residency laws (CON-30EA97016B). Cross-region replication will be disabled for PII-containing logs.

### 8.3 PII Segregation and Anonymization

To adhere to FTC guidelines on anonymity (CON-B3D71A437D) and ensure no de-anonymization attacks can link Beneficiaries to Donors:

1.  **Decoupled Logging:** Logs containing Beneficiary PII (e.g., eligibility checks) are stored in a separate, access-controlled log group from logs containing Donor PII (e.g., donation confirmations). These two log groups are never joined in a single query or dashboard.
2.  **Anonymized Analytics:** For donor impact receipts and redemption analytics (CON-23A501C051), logs must use UUIDv4 mapping to correlate events without linking PII. The mapping table itself is stored in a separate, highly restricted database (CON-0A0288EED4).
3.  **Access Control:** Access to logs containing PII is restricted to the Platform Administrator (ACT-086A974D63) and the Dispute Adjudicator ([ACT-7BA340FF76](../project_glossary.md#act-7ba340ff76)) on a need-to-know basis, with all access attempts logged to AWS CloudTrail ([CON-BB253DF0A2](../project_glossary.md#con-bb253df0a2)).

### 8.4 Compliance and Audit Trail

*   **SOC2 Type II Evidence:** All log archival and retention policies are automated via Infrastructure-as-Code (IaC) to ensure consistency and prevent manual tampering. Evidence of policy enforcement (e.g., successful archival jobs, masking rules applied) is logged to CloudTrail.
*   **Data Residency Compliance:** Log storage buckets are configured with region-specific tags. Automated compliance checks (e.g., AWS Config Rules) will alert if any PII-containing log is replicated outside its designated region.
*   **Audit Readiness:** The log aggregation system must support ad-hoc queries for compliance auditors, providing a complete, immutable trail of all financial transactions and administrative actions for the past 7 years.

### 8.5 Implementation Notes

*   **Ingestion Layer:** Use Fluent Bit or AWS CloudWatch Agent with custom parsing rules to apply masking before data is sent to CloudWatch or OpenSearch.
*   **Storage Layer:** Use S3 Object Lock with a Governance Mode to prevent deletion or modification of archived logs for the retention period.
*   **Access Layer:** Use AWS Lake Formation or OpenSearch fine-grained access control to restrict PII log access.
*   **Monitoring:** Set up alerts for any attempt to access PII logs outside of approved roles or regions.
