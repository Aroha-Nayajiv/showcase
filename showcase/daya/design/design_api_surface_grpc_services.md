# gRPC Service Contracts & Definitions

## 1. Service Boundaries and Responsibilities

The Payment Processing Surface ([SUR-5B18C8719F](../project_glossary.md#sur-5b18c8719f)) is decomposed into three primary gRPC services, each owning a distinct domain of the financial transaction lifecycle. These services enforce strict typing, ensure financial precision, and maintain the anonymity requirements mandated by the project's compliance goals.

1. **PaymentService**: The high-throughput, low-latency service responsible for real-time POS interactions. It handles the core "card tap" flow, converting a donor-funded credit into a merchant-accepted transaction. It must support streaming RPCs to provide real-time ledger updates to the POS terminal, addressing the latency constraint of Implied concern: Ensure Stripe Webhook Processing Latency averages below 150ms from card tap to merchant ledger entry. ([CON-06232374D9](../project_glossary.md#con-06232374d9)) and Implied concern: Latency optimization for real-time POS clearance to prevent restaurant queue stagnation ([CON-5D64EBC654](../project_glossary.md#con-5d64ebc654)).
2. **DisputeService**: The control plane for financial integrity. It manages the lifecycle of chargebacks, fraud investigations, and adjudication. It interfaces with the Platform-NGO Fraud Investigation Flow ([JNY-CA74D631DC](../project_glossary.md#jny-ca74d631dc)) and Beneficiary-Platform Dispute Flow ([JNY-2B038C9362](../project_glossary.md#jny-2b038c9362)) to ensure that disputes are resolved without exposing PII, adhering to Implied concern: Adhere to FTC guidelines on anonymity, ensuring no de-anonymization attacks can link beneficiaries to donors through metadata analysis ([CON-C22D030D21](../project_glossary.md#con-c22d030d21)).
3. **PayoutService**: The financial settlement service responsible for orchestrating payouts to Merchant Partners (Restaurants) via Stripe Connected Accounts. It handles the complexities of multi-jurisdictional KYC compliance (Implied concern: Manage Stripe Connected Account liability and KYC compliance across multiple jurisdictions (SF, NYC, Chicago) ([CON-5BFA25E8F9](../project_glossary.md#con-5bfa25e8f9))) and manages failure states, including Merchant Payout Failure & Error Handling ([CAP-MERCHANT-PAYOUT-FAILURE-ERROR-HANDLING](../project_glossary.md#cap-merchant-payout-failure-error-handling)).

## 4. Data Isolation and Anonymization

To comply with Implied concern: Implement strict data isolation where beneficiary demographic status and legal names are cryptographically segregated from public data (CON-0A0288EED4) and Implied concern: Implement strict data isolation where beneficiary demographic status and legal names are cryptographically segregated from public data (CON-92F07E31B0), the gRPC contracts enforce the following:

1. **No PII in Payment Flows**: The RedemptionRequest and RedemptionResponse messages do not contain any fields for beneficiary names, demographics, or legal identifiers. Only the beneficiary_token is used, which is a cryptographically secure, opaque reference.
2. **Hashed Identifiers**: Where beneficiary identification is necessary for dispute resolution, only a cryptographic hash of the beneficiary ID is transmitted, ensuring that PII is never exposed in the payment processing surface.
3. **Anonymized Analytics**: The TransactionRecord messages used for analytics and reporting (Implied concern: Correlate donor impact receipts with beneficiary redemption events without linking PII, using UUIDv4 mapping for analytics. (CON-23A501C051)) use UUIDv4 mapping to link donor impact to redemption events without exposing beneficiary PII.

## 5. Latency and Scalability

To meet the performance requirements of the platform:

1. **Low-Latency POS Clearance**: The PaymentService is designed to process transactions in under 150ms (Implied concern: Ensure Stripe Webhook Processing Latency averages below 150ms from card tap to merchant ledger entry. (CON-06232374D9)). This is achieved through the use of in-memory caching for token validation and asynchronous ledger writes.
2. **High Concurrency**: The PaymentService is designed to handle 10,000 concurrent connections (Implied concern: Maintain p99 latency below 250ms for voucher creation and scanning callbacks under 10,000 concurrent connections. ([CON-6D5E21557B](../project_glossary.md#con-6d5e21557b))). This is supported by the use of gRPC streaming and horizontal scaling of the service instances.
3. **Scalability**: The architecture is designed to scale from a single-city MVP to a multi-tenant architecture supporting 50,000 MAU across 3 metro footprints (Implied concern: Scalability of the anonymous credit distribution engine during peak event-driven load ([CON-121117F5A2](../project_glossary.md#con-121117f5a2))). The gRPC services are stateless and can be scaled independently based on load.

## 7. Integration with Sibling Artifacts

- **Financial Ledger Data Model**: The PaymentService and PayoutService interact with the Financial Ledger Data Model for all transaction records. The gRPC contracts define the interface for this interaction, but the detailed schema is owned by the Financial Ledger Data Model artifact.
- **User State & Profile Data Model**: The DisputeService interacts with the User State & Profile Data Model to retrieve beneficiary and merchant information for dispute resolution. The gRPC contracts use hashed identifiers to ensure PII is not exposed.
- **Stripe Issuing SDK Integration**: The PaymentService relies on the standard Stripe Issuing SDK for tokenization and card validation. The gRPC contracts define the interface for this interaction, but the detailed SDK integration logic is owned by the Stripe Issuing Proxy contract artifact.
- **Merchant Payout Failure & Error Handling**: The PayoutService implements the core payout logic, but the detailed error handling and retry workflows are owned by the Merchant Payout Failure & Error Handling artifact.

This artifact provides the complete gRPC service contracts for the Payment Processing Surface, enabling the Development phase to implement the core financial transaction flows of the MealCredit platform. The contracts are designed to be high-throughput, low-latency, and compliant with PCI-DSS Level 1 and FTC anonymity guidelines.

## 8. Overview and Scope

This artifact defines the gRPC service definitions, message schemas, and asynchronous transaction boundaries for the Payment Processing Surface of the MealCredit platform. It serves as the technical contract for the Transaction & Financial Engine, ensuring deterministic, low-latency, and PCI-DSS Level 1 compliant interactions between the Client Interface Layer, the API Orchestration Layer, and external financial rails.

The design strictly adheres to the project requirement to decouple food assistance from social stigma by converting real-time financial micro-donations into fractional, anonymous culinary credits. All service boundaries are designed to support the target scale of 50,000 MAU across the SF, NYC, and Chicago metropolitan footprints.

## 10. gRPC Error Handling and Status Codes

This section defines the standardized error handling contract for the Payment Processing Surface. It maps specific business logic failures—critical for maintaining financial integrity and PCI-DSS compliance—to appropriate gRPC status codes and structured `ErrorDetail` payloads. This ensures that the PaymentService, DisputeService, and PayoutService provide deterministic, actionable feedback to the Client Interface Layer and API Orchestration Layer.

### 10.1 ErrorDetail Message Structure

All gRPC errors MUST utilize the `ErrorDetail` message to provide structured, machine-readable context. This structure supports the platform's requirement for append-only cryptographic log auditing ([CON-1762EA5021](../project_glossary.md#con-1762ea5021)) and SOC2 Type II evidence collection ([CON-81FB01F06B](../project_glossary.md#con-81fb01f06b)) by ensuring every failure is logged with a consistent schema.

protobuf
message ErrorDetail {
  // Unique identifier for the specific error instance, used for tracing and audit logs.
  string error_id = 1;

  // A stable, machine-readable code for programmatic handling (e.g., INSUFFICIENT_CREDIT).
  string error_code = 2;

  // A human-readable message for logging or UI display (never expose raw PII).
  string message = 3;

  // Contextual metadata relevant to the failure (e.g., transaction_id, merchant_id, credit_pool_id).
  map<string, string> metadata = 4;

  // Timestamp of the error occurrence.
  google.protobuf.Timestamp timestamp = 5;
}

### 10.2 PaymentService Error Mapping

The PaymentService handles real-time POS taps and voucher redemptions. Errors here must be handled with low latency to prevent queue stagnation ([CON-4152F2C7C3](../project_glossary.md#con-4152f2c7c3), CON-5D64EBC654). Double-spending prevention (CON-61EC670500) is a critical failure mode that must be explicitly handled.

| Error Scenario | gRPC Status Code | Error Code | Description |
| :--- | :--- | :--- | :--- |
| Double-Spending Attempt | `FAILED_PRECONDITION` | `DOUBLE_SPEND_ATTEMPT` | The voucher token has already been redeemed or is currently being processed. The system must reject the request immediately to prevent ledger inconsistency. |
| Insufficient Credit Pool | `FAILED_PRECONDITION` | `INSUFFICIENT_CREDIT` | The Beneficiary's anonymous credit balance is less than the transaction amount. The POS terminal should prompt for alternative payment or partial redemption. |
| Expired Voucher Token | `INVALID_ARGUMENT` | `TOKEN_EXPIRED` | The time-bound cryptographic signature on the voucher has expired (CON-3335D67672). |
| Invalid Voucher Format | `INVALID_ARGUMENT` | `INVALID_TOKEN_FORMAT` | The voucher token does not match the expected cryptographic structure or UUIDv4 mapping (CON-23A501C051). |
| Merchant Not Found/Inactive | `NOT_FOUND` | `MERCHANT_NOT_FOUND` | The Merchant (ACT-AF904DCFF9) is not registered or has been suspended by the NGO Operator (ACT-09E028AEB0). |
| Geographic Restriction Violation | `PERMISSION_DENIED` | `GEO_RESTRICTION_VIOLATION` | The transaction is attempted outside the allowed metropolitan footprint (SF, NYC, Chicago) or the merchant's registered jurisdiction. |
| Rate Limit Exceeded | `RESOURCE_EXHAUSTED` | `RATE_LIMIT_EXCEEDED` | The Beneficiary or Merchant has exceeded the allowed number of transactions per minute to prevent abuse. |

### 10.5 Error Logging and Audit Trail

All errors MUST be logged to AWS CloudTrail ([CON-BB253DF0A2](../project_glossary.md#con-bb253df0a2), [CON-FBBBF07295](../project_glossary.md#con-fbbbf07295)) with the `error_id` and `error_code` to ensure traceability. The `metadata` field in `ErrorDetail` MUST NOT contain any raw PII or sensitive financial data (CON-0A0288EED4, CON-92F07E31B0, [CON-66390130AA](../project_glossary.md#con-66390130aa)). Instead, it should contain anonymized identifiers (e.g., hashed Beneficiary IDs, merchant IDs) to support analytics and dispute resolution without compromising anonymity ([CON-B3D71A437D](../project_glossary.md#con-b3d71a437d), CON-C22D030D21).

## 11. Asynchronous Transaction Boundaries and Event Schemas

This artifact defines the Protobuf message schemas and gRPC service boundaries for the asynchronous event streams within the Transaction & Financial Engine. These schemas facilitate the decoupling of high-throughput financial mutations from the synchronous Payment Processing Surface, ensuring the platform can handle 50,000 MAU across SF, NYC, and Chicago while maintaining strict PCI-DSS Level 1 and FTC anonymity compliance.

### 11.1 Event Stream Architecture and Boundaries

The Transaction & Financial Engine exposes three primary asynchronous event streams via gRPC Server-Sent Events (SSE) and Pub/Sub interfaces. These streams decouple the real-time POS clearing (PaymentService) from the financial ledger mutations and analytics.

1. **Donation-to-Redemption Velocity (DRV) Stream**: Monitors the liquidity health of the anonymous credit distribution engine. This stream emits events when the velocity of credit consumption approaches the 14-day target threshold ([CON-D0F5814F21](../project_glossary.md#con-d0f5814f21), [CON-F89C70071E](../project_glossary.md#con-f89c70071e)).
2. **Credit Pool Utilization Stream**: Provides real-time telemetry on the consumption of donor-funded credit pools. Automated alerts are triggered when utilization exceeds 85% ([CON-2059B17FB2](../project_glossary.md#con-2059b17fb2), [CON-7031BE57B3](../project_glossary.md#con-7031be57b3)).
3. **Stripe Webhook Latency Stream**: Tracks the end-to-end latency of Stripe Connected Account liability and KYC compliance events across jurisdictions (SF, NYC, Chicago) (CON-5BFA25E8F9, [CON-62097EBBF3](../project_glossary.md#con-62097ebbf3)). This ensures the average processing latency remains below 150ms from card tap to merchant ledger entry (CON-06232374D9, [CON-A0B785A40D](../project_glossary.md#con-a0b785a40d)).

### 11.2 Protobuf Message Definitions

The following Protobuf 3 definitions specify the exact structure of the asynchronous events. All monetary values are represented as strings to ensure exact precision and avoid floating-point errors in the financial ledger.

}

// DRVUpdateEvent is emitted when the Donation-to-Redemption Velocity (DRV) metric
// is calculated or when it approaches the 14-day liquidity target.
message DRVUpdateEvent {
  EventMetadata metadata = 1;
  string metro_footprint = 2; // e.g., "SF", "NYC", "CHI"
  double current_velocity = 3; // Credits consumed per day
  double target_velocity = 4; // 14-day target threshold
  double deviation_percentage = 5; // Percentage deviation from target
  string status = 6; // "NORMAL", "WARNING", "CRITICAL"
}

// CreditPoolUtilizationEvent is emitted when the Credit Pool Utilization Rate
// is updated, specifically triggering alerts when thresholds exceed 85%.
message CreditPoolUtilizationEvent {
  EventMetadata metadata = 1;
  string donor_funding_pool_id = 2; // UUIDv4 for the specific donor pool
  double utilization_rate = 3; // Percentage (0.0 to 1.0)
  double alert_threshold = 4; // e.g., 0.85 (CON-2059B17FB2)
  bool alert_triggered = 5; // True if utilization_rate > alert_threshold

// StripeWebhookLatencyEvent is emitted for each Stripe webhook processing cycle,

message StripeWebhookLatencyEvent {

### 12.1 Stripe Integration Contract (Payment Processing Surface)

The gRPC client for Stripe must strictly adhere to PCI-DSS Level 1 compliance by ensuring zero raw card data touches MealCredit servers. All financial transactions are routed through Stripe Elements and the Stripe SDK directly, as mandated by the project requirement. No internal proxy services are defined.

#### 6.1.1. gRPC Client Configuration
- **Service Endpoint**: `api.stripe.com:443` (Standard Stripe API)
- **Transport Security**: mTLS with mutual certificate validation for internal service-to-service communication; TLS 1.2+ for external Stripe API calls.
- **Timeouts**:
  - `connect_timeout`: 200ms
  - `request_timeout`: 150ms (aligned with CON-06232374D9: Ensure Stripe Webhook Processing Latency averages below 150ms from card tap to merchant ledger entry)
- **Retry Policy**: Exponential backoff with jitter for transient errors (5xx, rate limits). Max retries: 3.
- **Idempotency**: All `CreatePaymentIntent` and `CapturePayment` calls must include a unique `idempotency_key` (UUIDv4) to prevent double-spending (CON-61EC670500).

#### 6.1.2. Data Transformation Schema (Stripe -> MealCredit)
Stripe webhooks are transformed into internal gRPC events. PII is stripped at the boundary.

| Stripe Field | MealCredit Field | Transformation Logic | Compliance Note |
| :--- | :--- | :--- | :--- |
| `customer.id` | `beneficiary_hash` | SHA-256 hash of Stripe Customer ID | PII Segregation (CON-0A0288EED4) |
| `payment_intent.id` | `transaction_id` | Direct mapping (UUIDv4) | Traceability |
| `payment_intent.amount` | `transaction_amount` | Convert cents to decimal (USD) | Financial Precision |
| `payment_intent.status` | `transaction_status` | Map to PENDING, COMPLETED, FAILED | State Machine |
| `payment_intent.metadata.beneficiary_id` | `beneficiary_id` | Opaque string (Hash) | Anonymization (CON-B3D71A437D) |

#### 6.1.3. Error Handling & Edge Cases
- **Double-Spending Prevention**: The gRPC service must reject any `CapturePayment` request where the `idempotency_key` has already been used for a `COMPLETED` transaction.
- **Stripe Webhook Latency**: If the webhook processing latency exceeds 150ms, the system must trigger an alert (CON-06232374D9) and fallback to a cached status if available.

### 12.3 Redis Enterprise Cluster Integration Contract (Caching & State)

Redis is used for caching restaurant search queries ([CON-527BFA6796](../project_glossary.md#con-527bfa6796)) and storing temporary voucher states for low-latency POS clearing.

#### 6.3.1. Cache Key Structure
- **Voucher State**: `voucher:{voucher_id}:state` -> `{"status": "ACTIVE", "expires_at": "<timestamp>"}`
- **Merchant Geo-Index**: `merchant:geo:{lat}:{lon}:radius` -> `["merchant_id_1", "merchant_id_2"]`

#### 6.3.2. Cache Hit Ratio (CHR) Monitoring
- **Target**: Maintain Cache Hit Ratio (CHR) above 92% for restaurant search queries (CON-527BFA6796).
- **Monitoring**: The gRPC client must expose metrics for cache hits and misses to the Metrics Collection & Alerting Design artifact.

Description: Handles the core financial transactions, including voucher creation, redemption, and refund processing. This service acts as the proxy to Stripe Issuing for virtual card tokenization, utilizing the standard Stripe SDK directly as mandated by the project requirement.

service PaymentProcessingService {

  // Returns a tokenized credential that can be presented at a Merchant POS.

### 17.2 Dispute Resolution

1. **Trigger:** `PaymentProcessingService.SubmitDispute` is called by a Beneficiary.
2. **Synchronous Response:** The service returns a `SubmitDisputeResponse` with a dispute ID.
3. **Asynchronous Process:** The dispute is queued for review by the `Dispute Adjudicator` (ACT-7BA340FF76). The adjudicator reviews the immutable ledger logs and POS receipts.
4. **Notification:** Once resolved, the system updates the ledger and notifies both the Beneficiary and the Merchant.
