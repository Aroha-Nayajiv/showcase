# Offline Fallback & Accessibility Interface

This artifact defines the user interface and interaction flows for the offline fallback redemption mechanism in the MealCredit platform. It covers the Beneficiary's ability to generate and present offline tokens (via QR/barcode) when network connectivity is lost, and the Merchant's POS interface for scanning and validating these tokens. It also encompasses the accessibility requirements to ensure the interface is usable by individuals with low digital literacy and visual impairments.

### 1.1 Key UI Components

1. **Token Display Area:**
   - A large, high-contrast QR code centered on the screen.
   - The QR code must be generated using a time-bound cryptographic signature ([CON-3335D67672](../project_glossary.md#con-3335d67672)) to prevent replay attacks.
   - The QR code must not contain any PII or raw card data ([CON-66390130AA](../project_glossary.md#con-66390130aa)).

2. **Countdown Timer:**
   - A prominent, large-font countdown timer displayed below the QR code.
   - The timer indicates the remaining validity period of the current token.
   - The timer must update in real-time and change color when the token is about to expire.
   - The timer must be accessible via screen readers (VoiceOver/TalkBack) ([CON-68497304B1](../project_glossary.md#con-68497304b1)).

3. **Status Indicator:**
   - A clear status message indicating the current state: "Generating Token...", "Token Active", or "Token Expired. Refreshing...".
   - The status message must be large and high-contrast for low-vision users ([CON-6C177D0102](../project_glossary.md#con-6c177d0102)).

4. **Refresh Button:**
   - A large, easily tappable "Refresh Token" button displayed when the token expires or if the user manually requests a new token.
   - The button must have a clear label and sufficient touch target size (minimum 44x44 points).

5. **Error State:**
   - If SecureStore fails to retrieve the cryptographic signature ([CON-34312C6DC9](../project_glossary.md#con-34312c6dc9)), a clear, non-technical error message must be displayed.
   - The error message should include a "Retry" button and a "Contact Support" option.
   - The error message must be accessible via screen readers.

## 2. Merchant POS Offline Token Scanning Interface

This artifact defines the user interface and interaction flows for the Merchant ([ACT-AF904DCFF9](../project_glossary.md#act-af904dcff9)) to scan and validate offline fallback tokens generated by the Beneficiary ([ACT-ADA6716160](../project_glossary.md#act-ada6716160)). The interface is optimized for speed to prevent queue stagnation, aligning with latency concerns for POS clearance ([CON-4152F2C7C3](../project_glossary.md#con-4152f2c7c3), [CON-5D64EBC654](../project_glossary.md#con-5d64ebc654)). It includes the camera viewfinder overlay, real-time validation feedback (success/failure), and a manual entry fallback for cases where scanning fails.

### 2.1 Camera Viewfinder Overlay

The primary interaction mode is camera-based scanning. The viewfinder must be intuitive and accessible without requiring complex technical troubleshooting ([CON-387CDD9AEB](../project_glossary.md#con-387cdd9aeb)).

- **Visual Design:** A full-screen camera feed with a high-contrast, semi-transparent overlay. The active scanning zone is a centered square (or circle, depending on QR/barcode format) with a bright, pulsing border to guide the user.
- **Feedback:** Haptic feedback (vibration) upon successful detection of a QR/barcode pattern, even before cryptographic validation is complete. This provides immediate confirmation that the token was read.
- **Accessibility:** The viewfinder must be fully compatible with screen readers and high-contrast modes for visually impaired users (CON-68497304B1). VoiceOver/TalkBack announcements should include the current scanning status and any error messages.
- **Performance:** The viewfinder must maintain a high frame rate to ensure quick detection. The scanning logic should run on a background thread to prevent UI jank.

### 2.2 Real-Time Validation Feedback

Once a token is scanned, the interface must provide immediate, clear feedback on its validity. This is critical for maintaining queue flow.

- **Success State:** Upon successful validation, the screen transitions to a full-screen green overlay with a large checkmark icon and a clear "Payment Accepted" message. The token amount and any relevant details (e.g., "MealCredit Applied") are displayed. A brief, positive haptic pulse confirms the transaction.
- **Failure State:** If the token is invalid, expired, or already used, the screen transitions to a full-screen red overlay with a clear "Invalid Token" message and a brief explanation (e.g., "Token Expired" or "Token Already Redeemed"). The user is prompted to ask the Beneficiary to generate a new token or use the manual entry fallback.
- **Latency:** The validation feedback must be displayed within the target latency threshold established by the Merchant POS Integration & Dashboard artifact to prevent queue stagnation.

### 2.3 Manual Entry Fallback

In cases where scanning fails (e.g., damaged QR code, poor lighting), a manual entry fallback must be available.

- **Access:** A clearly visible "Enter Code Manually" button is placed at the bottom of the viewfinder screen. This button must be easily accessible and tappable.
- **Input Field:** Tapping the button opens a modal or slide-up panel with a single text input field. The field should support auto-capitalization and auto-correction for alphanumeric codes.
- **Validation:** The same validation logic used for scanned tokens is applied to manually entered codes. Feedback (success/failure) is displayed in the same manner as for scanned tokens.
- **Accessibility:** The manual entry interface must be fully navigable using keyboard-only navigation and low-vision readability standards (CON-6C177D0102). Screen readers must announce the input field placeholder and any validation errors.

### 2.4 Error Handling and Edge Cases

The interface must handle various error states gracefully to ensure a smooth user experience.

- **Network Unavailable:** If the device is offline, the interface should still allow scanning and local validation of time-bound tokens. If the token requires online verification (e.g., for double-spend prevention), a clear message should be displayed, and the transaction should be queued for later synchronization.
- **Camera Permissions:** If camera permissions are denied, the interface should display a clear message explaining why scanning is unavailable and prompt the user to enable permissions in their device settings. The manual entry fallback should remain available.
- **Invalid Token Format:** If the scanned or entered code does not match the expected format, a clear error message should be displayed, and the user should be prompted to try again.

### 2.5 Accessibility Compliance

The entire interface must adhere to WCAG 2.1 AA standards to ensure accessibility for all users.

- **Color Contrast:** All text and interactive elements must have a minimum contrast ratio of 4.5:1 against their background.
- **Font Size:** Text must be scalable up to 200% without loss of content or functionality.
- **Touch Targets:** All interactive elements must have a minimum touch target size of 44x44 points.
- **Screen Reader Support:** All interactive elements must have appropriate ARIA labels and roles. Dynamic content (e.g., validation feedback) must be announced to screen readers using live regions.

### 2.6 Testing Strategy

- Unit tests for the validation logic and error handling.
- Integration tests for the camera viewfinder and manual entry flow.
- Accessibility tests using VoiceOver and TalkBack.
- Performance tests to ensure the viewfinder maintains a high frame rate and validation feedback is displayed within the target latency.
- Security tests to ensure no PII or raw card data is exposed.

### 3.1 Screen Reader Compatibility (VoiceOver/TalkBack)

- **QR Code Accessibility:** The QR code itself is not readable by screen readers. It must be accompanied by a dynamic text label that announces the token's status and remaining validity time. For example: "Offline Token Active. Valid for 4 minutes."
- **Text-to-Speech (TTS) Option:** A prominent "Listen to Token Code" button must be provided. When activated, the app should use the device's native TTS engine to read the alphanumeric fallback code (if applicable) or the token's unique identifier. This ensures users who cannot scan the QR code can still present the token to a merchant.
- **Dynamic Announcements:** Use AccessibilityInfo and AccessibilityEvent to announce state changes. If the token is about to expire (e.g., < 30 seconds), a high-priority announcement should trigger: "Token expiring soon. Please refresh."

### 3.2 High-Contrast and Visual Adaptations

- **Color Palette:** The interface must support a high-contrast mode. Primary action elements (e.g., "Refresh Token") must have a contrast ratio of at least 4.5:1 against their background. In high-contrast mode, borders and outlines should be thickened (minimum 2px) to ensure visibility.
- **Typography:** Use scalable font sizes. The base font size should be 16pt, with support for dynamic type up to 200% scaling without layout breakage. The token validity countdown timer must use a monospaced font to prevent jitter as numbers change.
- **Visual Cues:** In addition to color, use icons and text labels for all interactive elements. For example, a "Refresh" icon should always be accompanied by the text "Refresh".

### 3.3 Error and Empty States

- **SecureStore Failure:** If the offline token cannot be generated due to a SecureStore error (CON-34312C6DC9), the UI must display a clear, non-technical error message: "Unable to generate offline token. Please check your device settings and try again." A "Retry" button must be prominently displayed.
- **Network Unavailable:** If the app detects no network connectivity, it should automatically switch to the offline fallback view. A subtle banner at the top should indicate: "Offline Mode Active."

## 4. Merchant POS Dashboard (Next.js/Web)

The Merchant POS interface must be fully navigable via keyboard to support operators who may not use a mouse or touchscreen.

### 4.1 Keyboard Navigation

- **Tab Order:** All interactive elements (scan button, manual entry field, validation status) must be focusable in a logical order. The primary action (e.g., "Scan Token") should receive initial focus.
- **Focus Indicators:** A visible, high-contrast focus ring (minimum 2px) must be applied to all focusable elements. The focus ring should not be removed via CSS (`outline: none`) without a suitable alternative.
- **Keyboard Shortcuts:** Implement keyboard shortcuts for common actions. For example, `Ctrl+Enter` or `Cmd+Enter` should trigger the "Scan Token" action. Escape should clear the current validation state.

### 4.2 Visual Clarity for POS Environment

- **High-Contrast Mode:** The POS dashboard must support a high-contrast theme optimized for bright, well-lit restaurant environments. Text and icons must remain legible against the background.
- **Large Touch Targets:** All buttons and interactive elements must have a minimum touch target size of 44x44 pixels to accommodate operators wearing gloves or working quickly.

## 9. Offline Fallback Redemption User Flows

This document defines the sequential user journeys for the Offline Fallback Redemption process within the MealCredit platform. These flows address the critical requirement to decouple food assistance from social stigma ([CON-B3D71A437D](../project_glossary.md#con-b3d71a437d)) while ensuring transaction integrity without real-time network connectivity. The flows are designed for the Expo mobile application (Beneficiary) and the Merchant POS interface, strictly adhering to WCAG 2.1 AA standards (CON-68497304B1) and PCI-DSS Level 1 compliance (CON-66390130AA) by ensuring no raw card data or PII is exposed during the offline token exchange.

### 9.1 Beneficiary Journey: Token Generation and Presentation

This flow details the experience for the Beneficiary (ACT-ADA6716160) when network connectivity is lost, transitioning from online voucher status to offline token generation.

1. **Connectivity Loss Detection**
   - **Trigger:** The Expo mobile application detects a loss of network connectivity (Wi-Fi/Cellular) while the Beneficiary is attempting to access their voucher or is already in the redemption view.
   - **System Action:** The app immediately switches to an offline-capable state. It retrieves the latest cached voucher data from the local SecureStore (CON-34312C6DC9).
   - **UI Feedback:** A non-intrusive banner appears at the top of the screen: "Offline Mode: Using secure offline token."
   - **Accessibility:** Screen readers announce "Offline mode active. Your meal credit is available."

2. **Offline Token Generation**
   - **Action:** The app generates a time-bound, cryptographically signed QR/barcode token. This token is derived from the cached voucher data and a local secret key, ensuring it cannot be replayed after expiration (CON-3335D67672).
   - **Security:** No PII (legal names, demographic status) is included in the QR code payload. Only a UUIDv4 mapping and a cryptographic signature are used ([CON-23A501C051](../project_glossary.md#con-23a501c051), [CON-92F07E31B0](../project_glossary.md#con-92f07e31b0)).
   - **UI Feedback:** A large, high-contrast QR code is displayed prominently. A countdown timer indicates the token's remaining validity (e.g., "Valid for 5:00").
   - **Accessibility:** The QR code is accompanied by a large, dynamic text label. The countdown timer is announced periodically by screen readers. High-contrast mode is automatically enabled for the QR code area (CON-68497304B1).

3. **Token Presentation**
   - **Action:** The Beneficiary presents the QR code to the Merchant (ACT-AF904DCFF9).
   - **UI Feedback:** The screen remains active, preventing sleep mode during the transaction. The timer continues to count down.
   - **Edge Case - Token Expiration:** If the timer reaches zero before the Merchant scans the token, the screen displays a "Token Expired" message with a "Refresh Token" button. The user must tap to generate a new token.

4. **Post-Transaction Synchronization**
   - **Action:** Once connectivity is restored, the app automatically synchronizes the offline transaction with the central ledger.
   - **UI Feedback:** A subtle success animation appears: "Transaction synced."
   - **Error Handling:** If synchronization fails, the app queues the transaction and notifies the user: "Transaction saved. Will sync when online."

### 9.2 Merchant Journey: Scanning and Validation

This flow details the experience for the Merchant (ACT-AF904DCFF9) when scanning an offline token via the POS interface. The Merchant POS integration is handled by a sibling artifact; this flow focuses on the validation logic and user feedback specific to offline tokens.

1. **Scan Initiation**
   - **Action:** The Merchant opens the "Scan Offline Token" mode in the POS interface.
   - **UI Feedback:** The camera viewfinder activates with a clear overlay indicating "Scan Offline MealCredit Token."
   - **Accessibility:** Voice prompts guide the Merchant: "Camera active. Scan the QR code."

2. **Token Decryption and Validation**
   - **Action:** The POS system decrypts the QR code payload using the shared public key. It validates the cryptographic signature and checks the timestamp against the local device clock (since network is unavailable).
   - **Security:** The system verifies that the token has not been previously used (checking against a local cache of recently scanned tokens to prevent replay attacks, CON-3335D67672).
   - **Edge Case - Invalid Signature:** If the signature is invalid, the screen displays a red "Invalid Token" message with a "Try Again" button. No PII is displayed.
   - **Edge Case - Expired Token:** If the token's timestamp is outside the valid window, the screen displays "Token Expired" with instructions for the Beneficiary to refresh.

3. **Validation Success and Amount Confirmation**
   - **Action:** If the token is valid, the POS system displays the transaction amount and the remaining credit balance (anonymized).
   - **UI Feedback:** A green checkmark appears with the message "Token Valid. Amount: $X.XX."
   - **Accessibility:** Screen readers announce "Token valid. Amount is X dollars."

4. **Transaction Completion**
   - **Action:** The Merchant confirms the transaction. The POS system records the transaction locally with a unique ID and a timestamp.
   - **UI Feedback:** The screen displays "Transaction Complete."
   - **Post-Transaction:** The POS system queues the transaction for synchronization with the central ledger once connectivity is restored.

  - **Why Critical:** This determines the security window for the time-bound token and impacts the user experience if clocks are significantly out of sync.
  - **Answerable:** No - requires security and product team input.

### 9.7 Validation Criteria

- **Functional:** The Beneficiary can generate and present an offline token. The Merchant can scan and validate the token. Transactions are queued for sync.
- **Security:** The offline token is cryptographically signed and time-bound. No PII is exposed. Replay attacks are prevented.
- **Accessibility:** The interface meets WCAG 2.1 AA standards. Screen readers and keyboard navigation are fully supported.
- **Error Handling:** All edge cases (expiration, invalid signature, double-spend) are handled gracefully with clear user feedback.

### 9.8 Conclusion

This document provides a comprehensive definition of the offline fallback redemption user flows for the MealCredit platform. By addressing the journeys of both the Beneficiary and the Merchant, and by incorporating robust security, accessibility, and error handling measures, these flows ensure a seamless and dignified experience for all users, even in the absence of network connectivity. The flows are designed to be implemented directly by the development team, with clear guidance on UI/UX, security, and accessibility requirements.
