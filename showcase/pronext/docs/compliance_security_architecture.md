# Compliance and Security Architecture Foundations

## 1. Executive Summary and Scope

This artifact establishes the binding compliance and security architecture for ProNext, a unified AI-native healthcare procurement platform. It maps explicit regulatory obligations (HIPAA, HITRUST, SOC2) to the system's actual concern surface, ensuring that the 'Whole Product' vision respects the strictest compliance constraints from day one. The artifact defines the 'Data Perimeter'—the hard boundary ensuring zero data egress—and establishes 'Agent Governance' policies for autonomous procurement workflows.

This document serves as the authoritative reference for the `Data Perimeter and AI Inference Constraints` and `Agent Governance and Human-in-the-Loop Confirmation` capabilities. It explicitly excludes integration mechanics (e.g., specific ERP connector protocols) and performance SLAs, which are reserved for the `Operational Constraints and Integration Requirements` artifact.

## 2. Regulatory Mapping and Control Objectives

The 'Zero Data Egress' rule is not merely a technical preference but a regulatory mandate derived from the following frameworks. This mapping ensures that ProNext's AI-native capabilities (Intelligent Ordering, Agentic Automation) do not violate core compliance tenets.

| Framework | Control Reference | Control Objective | ProNext Application |
| :--- | :--- | :--- | :--- |
| **HIPAA** | 45 CFR § 164.312(a)(1) - Access Control | Implement policies/procedures for granting, reviewing, and revoking access to ePHI. | Egress of ePHI to external, uncontrolled AI endpoints violates controlled access. All inference must remain within the client perimeter. |
| **HITRUST** | CSF v9.0 - DLP-01 | Mandate Data Loss Prevention controls to ensure sensitive data is not transmitted outside the security perimeter without explicit authorization. | The AI Runtime is treated as an internal data processing component. No inference payloads may be sent to external APIs (e.g., OpenAI, AWS Bedrock). |
| **SOC 2** | CC6.1 - Logical and Physical Access Controls | Implement logical access security software, infrastructure, and architectures over protected information assets. | Allowing inference data to leave the perimeter breaks the chain of custody and auditability required for SOC 2 certification. |

## 3. The Zero Data Egress Boundary

The 'Zero Data Egress' rule establishes a hard architectural boundary. The following data types are strictly prohibited from leaving the client's infrastructure perimeter (VPC, on-prem cluster, or dedicated tenant environment):

### 3.1 Prohibited Data Types

1. **Model Weights**: All LLM and specialized procurement AI model weights must be stored in a private, air-gapped registry within the client perimeter. No weights may be downloaded from external public registries (e.g., Hugging Face, Docker Hub) without a strict, audited staging and signing process.
2. **Inference Data**: All prompts, context windows, and intermediate reasoning steps generated during the `IntelligentOrdering` and `AgenticProcurementAutomation` journeys must be processed locally. No inference payloads may be sent to external API endpoints for processing.
3. **Agent Decision Logs**: All logs generated by AI agents, including low-stock detection triggers, vendor selection rationale, and requisition pre-filling data, must be stored in client-owned databases. These logs are critical for audit trails and must not be transmitted to external telemetry services by default.

### 3.2 Operational Constraints for the AI Runtime

To enforce this rule, the AI Runtime must adhere to the following operational constraints:

*   **Local Deployment**: The AI Runtime must be deployed as a sidecar or co-located service within the client's infrastructure. It must not rely on external cloud-based LLM APIs for real-time inference.
*   **Network Isolation**: The AI Runtime must operate in a network segment with no outbound internet access, except for explicitly whitelisted, authenticated endpoints for model updates.
*   **Encryption at Rest and in Transit**: All model weights and inference data must be encrypted using AES-256 or equivalent standards. Keys must be managed within the client's KMS (Key Management Service) or HSM (Hardware Security Module).
*   **Telemetry Disablement**: All outbound telemetry, error reporting, and usage analytics must be disabled by default. If enabled, they must be strictly filtered to exclude any PHI, PII, or sensitive procurement data.

## 4. Agent Governance and Human-in-the-Loop Confirmation

ProNext's AI agents operate with high autonomy but are bound by strict governance policies to prevent financial and compliance drift. This section defines the governance framework for autonomous procurement workflows.

### 4.1 Mandatory Human-in-the-Loop (HITL) Gate

A mandatory human-in-the-loop confirmation gate is required before any financial commitment is made by AI agents. This is a non-negotiable business rule derived from the SoftwareDNA.

*   **Trigger**: When an AI agent identifies low stock, selects a vendor, and pre-fills a requisition with negotiated pricing.
*   **Action**: The requisition is routed through the correct approval chain. The `Finance Approver` (ACT-D0B70FB938) must explicitly confirm the approval.
*   **Outcome**: Only upon explicit confirmation is the requisition converted to a Purchase Order (PO). The `RequisitiontoOrderConversion` journey (JNY-9A5F9E3C00) is blocked until this gate is passed.

### 4.2 Maverick Spend Prevention

AI must flag cheaper compliant alternatives and prevent maverick spend before order submission, not after. This is enforced by the `IntelligentOrdering` capability, which checks GPO contract rates and contract compliance during the ordering process.

### 4.3 Audit Trails and Evidence Requirements

To satisfy SOC2 and HITRUST certification requirements, the following evidence must be captured and retained for AI-driven decisions:

1.  **Network Flow Diagrams**: Documented network flows showing no outbound connections from the AI Runtime to external IPs, except for the whitelisted update endpoints.
2.  **Access Logs**: Logs demonstrating that only authorized personnel (e.g., `System Operator` (ACT-0FC1969E12)) can access model weights and inference data.
3.  **Encryption Verification**: Evidence that all data at rest and in transit is encrypted using approved algorithms.
4.  **Telemetry Configuration**: Configuration files or screenshots showing that outbound telemetry is disabled or strictly filtered.
5.  **Agent Decision Logs**: Immutable logs of AI reasoning, vendor selection rationale, and price variance flags, retained for the period required by regulatory audit standards.

## 5. Model Update and Maintenance Protocol

While inference is strictly local, model updates are necessary for maintaining accuracy and compliance. The following protocol governs model updates:

1.  **Staging Area**: New model weights must be downloaded to a secure, isolated staging area within the client perimeter.
2.  **Validation**: Models must undergo a validation process to ensure they have not been tampered with and are compatible with the current runtime environment.
3.  **Approval**: Model updates require explicit approval from the `System Operator` (ACT-0FC1969E12) before being deployed to the production AI Runtime.
4.  **Audit Trail**: All model update activities, including download, validation, and deployment, must be logged and retained for audit purposes.
