# Operational Constraints and Integration Requirements

**Artifact ID:** inception_operational_constraints
**Owner:** System Operator (ACT-0FC1969E12)
**Status:** Ratified Draft

## 1. Executive Summary and Scope

This artifact establishes the non-negotiable operational boundaries for the ProNext platform. It defines the constraints governing the 'Zero Data Egress' mandate for AI inference, the integration contracts for cXML/OCI PunchOut with major suppliers, and the data model invariants required for a unified procurement lifecycle across diverse care settings (SNFs, ALFs, Rehab Centers).

This document serves as the binding operational contract for the Design and Implementation phases. It explicitly defers detailed security architecture controls (e.g., specific encryption key lengths, HSM configurations) to the sibling artifact inception_compliance_security_architecture and focuses strictly on the operational enforcement mechanisms and integration interfaces.

## 2. Zero Data Egress Mandate

**Constraint ID:** OPR-CONST-001
**Scope:** All AI inference, model training, and agent decision logging.
**Rule:** Under no circumstances shall any data classified as Protected Health Information (PHI), Personally Identifiable Information (PII), or sensitive financial data be transmitted to external AI model providers, third-party telemetry services, or public cloud inference endpoints.

### 2.1 Local Runtime Deployment
The AI Runtime must be deployed as a containerized service within the client's Virtual Private Cloud (VPC) or on-premises cluster. This ensures that all data processing occurs within the trusted boundary defined by the client's infrastructure perimeter.

### 2.2 Model Weights Isolation
All model weights, including those for natural language understanding and procurement optimization, must be stored in a private, air-gapped registry accessible only from within the client perimeter. No model weights may be pulled from public registries without a verified, signed checksum and internal security review.

### 2.3 Inference Data Containment
Inference requests and responses must never leave the client network. The AI Runtime must utilize local models that are fine-tuned or prompted to operate without external API calls for data processing.

**Decision Gap:** The specific local/open-source models to be deployed (e.g., Llama 3, Mistral) and the fine-tuning strategy remain unresolved. This decision must be ratified by the System Operator (ACT-0FC1969E12) prior to the Design Phase.

### 2.4 Agent Decision Logs
All logs generated by the Agentic Procurement Automation (e.g., low-stock detection, vendor selection, requisition pre-filling) must be stored in encrypted, client-owned databases. These logs must not be exported for model improvement or telemetry without explicit, anonymized, and consented data governance approval.

## 3. Integration Standards for External Systems

**Constraint ID:** OPR-CONST-002
**Scope:** Supplier PunchOut and ERP/EHR Integration.
**Rule:** ProNext must support standardized, secure integration protocols for external data exchange, ensuring data integrity and compliance during transit.

### 3.1 Supplier Integration (PunchOut)
ProNext must support bidirectional, standards-compliant integration with major healthcare suppliers (McKesson, Medline, Cardinal Health, Sysco) using cXML and OCI PunchOut protocols. These contracts ensure that the AI agent can seamlessly access supplier catalogs, enforce GPO-negotiated pricing, and submit orders without manual intervention.

*   **Protocol Support:** Support for cXML and OCI PunchOut standards for direct supplier shopping.
*   **Security:** All PunchOut sessions must be secured via TLS 1.3. Authentication must use OAuth 2.0 or SAML 2.0, depending on supplier capabilities.
*   **Data Minimization:** Only the minimum necessary data (e.g., item SKUs, quantities, pricing) should be exchanged. No PHI should be included in PunchOut requests or responses.

### 3.2 ERP/EHR Integration
*   **Protocol:** Support for EDI 810 (Invoice) and EDI 850 (Purchase Order) for legacy ERP systems. For modern ERPs, RESTful APIs with OAuth 2.0 authentication must be used.
*   **Data Mapping:** A canonical data model must be maintained within ProNext to map external ERP/EHR data fields to internal ProNext entities, ensuring consistency across diverse care settings.
*   **Error Handling:** Integration failures must be logged locally and trigger alerts to the System Operator (ACT-0FC1969E12) without exposing sensitive data in error messages.

## 4. Compliance and Auditability

**Constraint ID:** OPR-CONST-003
**Scope:** HIPAA, HITRUST, and SOC2 Compliance.
**Rule:** All operational constraints must be auditable and demonstrable to compliance officers.

### 4.1 Audit Trail
*   **Immutable Logs:** All data access, inference requests, and agent actions must be recorded in an immutable audit log. These logs must include timestamps, user IDs (e.g., Procurement Requester, Finance Approver), and action types.
*   **Retention:** Audit logs must be retained for a minimum period as per HIPAA requirements, and stored in a write-once, read-many (WORM) storage format.
*   **Access Control:** Access to audit logs must be restricted to authorized System Operators (ACT-0FC1969E12). Any access to audit logs must itself be logged.

### 4.2 Data Encryption
*   **At Rest:** All data, including model weights, inference data, and logs, must be encrypted using strong encryption standards.
*   **In Transit:** All data in transit, whether internal or external, must be encrypted using TLS 1.3.

## 5. Validation Criteria

To ensure compliance with the Zero Data Egress mandate and integration standards, the following validation criteria must be met prior to Design Phase completion:

1.  **Network Flow Diagrams:** Must show no outbound connections from the AI Runtime to external IPs except for explicitly whitelisted, authenticated update endpoints.
2.  **Penetration Testing:** Must confirm no data exfiltration via DNS, HTTP, or other protocols.
3.  **Audit Logs:** Must demonstrate that all model weights and inference data remain within the defined storage buckets/volumes.
4.  **Code Review:** Must confirm no hardcoded external endpoints or unencrypted data transmission.

## 8. Data Model Invariants

To ensure consistency across diverse care settings, the following data model invariants must be maintained:

1.  **Unified Item Catalog:** All items must be mapped to a unified catalog with standardized SKUs and descriptions.
2.  **Facility Hierarchy:** A clear hierarchy of facilities, departments, and cost centers must be maintained.
3.  **Vendor Master Data:** A single source of truth for vendor information, including GPO contract details, must be maintained.
4.  **User Roles and Permissions:** A consistent RBAC model must be enforced across all modules.
